
Nguyên tắc cốt lõi: **container luôn listen port cố định, chỉ thay đổi mapping bên ngoài.**

**Dockerfile** — hardcode port, không bao giờ dùng biến:

```dockerfile
# ✅ Port cố định trong container
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

# ❌ Đừng làm thế này
EXPOSE ${APP_PORT}
CMD ["uvicorn", "app.main:app", "--port", "${APP_PORT}"]
```

Container là immutable artifact — port bên trong không có lý do thay đổi. `EXPOSE` là documentation, không ảnh hưởng runtime.

**`.env`** — chỉ định nghĩa host port (port bên ngoài):

```env
# ✅ Chỉ host port mapping
LESSON_API_PORT=30001

# ❌ Không cần cái này
CONTAINER_PORT=8000
```

**`docker-compose.yml`** — map host port (biến) → container port (cố định):

```yaml
services:
  lesson-api:
    build: .
    ports:
      - "${LESSON_API_PORT:-30001}:8000"   # host:container
    # ✅ host port thay đổi theo env
    # ✅ container port luôn 8000
```

**Tại sao làm vậy:**

Container port cố định vì nó là internal network — các service giao tiếp với nhau qua service name + container port (`http://lesson-api:8000`), không cần biết host port. Host port mới là thứ cần linh hoạt vì trên cùng một server có thể chạy nhiều instance hoặc nhiều service cùng loại.

Nếu để container port là biến, bạn tạo ra một layer indirection không cần thiết — ai đó đổi `APP_PORT` trong `.env` nhưng quên rebuild image, hoặc hai file `.env` khác nhau giữa dev/staging gây ra bug khó trace.

**Ví dụ thực tế cho Pika stack:**

```env
# .env
LESSON_API_PORT=30001
EMOTION_SERVICE_PORT=30002
ORCHESTRATION_PORT=30003
```

```yaml
# docker-compose.yml
services:
  lesson-api:
    build: ./lesson-api
    ports:
      - "${LESSON_API_PORT:-30001}:8000"

  emotion-service:
    build: ./emotion-service
    ports:
      - "${EMOTION_SERVICE_PORT:-30002}:8000"

  orchestration:
    build: ./orchestration
    ports:
      - "${ORCHESTRATION_PORT:-30003}:8000"
```

Mọi container đều listen `8000` bên trong. Service-to-service gọi `http://emotion-service:8000`. Chỉ host port khác nhau để expose ra ngoài.

---

Cả hai đều cần, nhưng cho mục đích khác nhau:

|Directive|Purpose|
|---|---|
|`ports: "30002:8000"`|**External access** - Map host port 30002 → container port 8000|
|`env_file: .env`|**Internal config** - Load env vars vào trong container cho app đọc|

```yaml
ports:
  - "30002:8000"    # Host:30002 → Container:8000
env_file:
  - .env            # Load biến môi trường vào trong container
```

Khi request từ bên ngoài:

1. Request đến `localhost:30002` (host port)
2. Docker forward đến container port `8000`
3. Uvicorn listen trên `8000` trong container
4. App đọc config từ env variables (từ .env)

Cả hai đều cần thiết - `ports` cho network, `env_file` cho configuration.

```bash
# Project name riêng cho API service
name: context-handling-service

services:
  api:
    build:
      context: .
      dockerfile: ./Dockerfile
    container_name: context_handling_api
    restart: unless-stopped
    env_file:
      - .env

      LOGGING_LEVEL: INFO
      ENVIRONMENT: production
      HOST: 0.0.0.0
      # ✅ Uvicorn Workers Configuration
      # Số workers trong 1 container (multi-process)
      # Default: 4 workers nếu không set
      UVICORN_WORKERS: ${UVICORN_WORKERS:-4}
      # ✅ Database Pool Size (per worker)
      # Công thức: DB_POOL_SIZE per worker = 10-15 connections
      # Total connections = UVICORN_WORKERS × DB_POOL_SIZE
      # Với 4 workers: 4 × 15 = 60 connections (an toàn)
      # Giảm từ 50 xuống 15 để tránh vượt quá PostgreSQL max_connections
      DB_POOL_SIZE: ${DB_POOL_SIZE:-15}
      DB_MAX_OVERFLOW: ${DB_MAX_OVERFLOW:-10}
    ports:
      - "${PORT-30020}:30020"
    networks:
      - context_handling_network
    # ✅ Tăng memory và CPU limits vì có nhiều workers
    deploy:
      resources:
        limits:
          memory: 3G  # Tăng từ 2G lên 3G (4 workers × ~500MB = 2GB)
          cpus: '4.0'  # Tăng từ 2.0 lên 4.0 để tận dụng 4 workers
        reservations:
          memory: 1G
          cpus: '1.0'

# Networks - Use external network (created by infrastructure)
networks:
  context_handling_network:
    external: true
    name: context_handling_network

```

Câu hỏi hay! Đây là 2 khái niệm khác nhau:

## 1. **Ports** (docker-compose)

→ Xác định **bên ngoài** container access vào service như thế nào

```yaml
ports:
  - "8000:8000"  # host:container
```

→ Từ máy host gọi `localhost:8000` sẽ forward vào container port 8000

## 2. **Environment Variables** (.env)

→ Xác định **bên trong** container: app cần biết những gì để chạy?

|Environment Variable|Mục đích|
|---|---|
|`DATABASE_URL`|App kết nối DB bằng gì?|
|`REDIS_URL`|App kết nối Redis bằng gì?|
|`LOG_LEVEL`|App log level là gì?|
|`SECRET_KEY`|JWT signing key|
|`RABBITMQ_URL`|App kết nối RabbitMQ bằng gì?|

## Ví dụ cụ thể

```yaml
# docker-compose.prod.yml
services:
  api:
    ports:
      - "8000:8000"  # ✅ Port để access từ outside
    
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb  # ⚠️ Cần vì app cần biết kết nối DB
      - REDIS_URL=redis://cache:6379
```

- **Port 8000**: Người dùng gọi API từ browser/frontend
- **DATABASE_URL**: App bên trong container cần để kết nối PostgreSQL (db là hostname nội bộ của container khác)

## Tóm lại

||Ports|Environment Variables|
|---|---|---|
|**Ai dùng**|Client bên ngoài|App bên trong container|
|**Mục đích**|Access service|Configure app behavior|
|**Cần thiết?**|Để expose ra ngoài|Để app chạy được|

**两者缺一不可** - Port để người dùng gọi được, env để app biết cách hoạt động.

Ý mình là sao đã có full .env rồi mà vẫn cần phải enviroment trong docker compose @src/docker-compose.prod.yml

À, câu hỏi của bạn là: **đã có `.env` đầy đủ rồi, tại sao vẫn cần khai báo thêm `environment:` trong docker-compose?**

## Thực ra... có thể **không cần** thiết!

Trong trường hợp của bạn, nếu `.env` đã có đầy đủ các biến này:

```env
LOGGING_LEVEL=INFO
ENVIRONMENT=production
HOST=0.0.0.0
UVICORN_WORKERS=4
DB_POOL_SIZE=15
DB_MAX_OVERFLOW=10
PORT=30020
```

Thì phần `environment:` trong docker-compose là **redundant (thừa)** - trừ khi:

### 1. Muốn override giá trị từ .env (không nên)

```yaml
environment:
  - ENVIRONMENT=production  # OK - đảm bảo luôn là production
  - DB_POOL_SIZE=15         # Thừa - đã có trong .env
```

### 2. Set giá trị mặc định an toàn

```yaml
environment:
  - UVICORN_WORKERS=${UVICORN_WORKERS:-4}  # Dùng default nếu .env không có
```

### 3. Biến infrastructure-specific

```yaml
environment:
  - HOST=0.0.0.0  # Container cần biết listen ở đâu
```

## Đề xuất clean hơn

Nếu `.env` đã đầy đủ, có thể simplify:

```yaml
services:
  api:
    build:
      context: .
      dockerfile: ./Dockerfile
    container_name: context_handling_api
    restart: unless-stopped
    
    # ✅ Dùng .env là đủ
    env_file:
      - .env
    
    # Chỉ cần expose port
    ports:
      - "${PORT:-30020}:30020"
    
    networks:
      - context_handling_network
```

**Kết luận**: `environment:` trong docker-compose là **optional** - dùng khi cần override hoặc set default. Nếu `.env` đã đầy đủ thì không cần thiết. Bạn có thể clean up!
