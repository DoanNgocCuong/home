## I. ARCHITECTURE & DESIGN

### I.1 — Monolithic architecture cho hệ thống lớn, không có separation of concerns

|                                      | Chi tiết                                                                                                                                                                                                              |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Deploy 1 tính năng nhỏ phải deploy toàn bộ. 1 lỗi nhỏ làm sập toàn hệ thống. Team conflict khi merge code.                                                                                                |
| **Tác động**                | Downtime toàn hệ thống khi deploy. Không thể scale từng phần độc lập. Velocity giảm dần theo thời gian.                                                                                                   |
| **Best Practice**              | Bắt đầu với**Modular Monolith** , tách module rõ ràng. Chỉ chuyển sang Microservices khi team > 10 người và domain boundaries rõ. Áp dụng**Strangler Fig Pattern**để migration dần dần. |

### I.2 — Chọn sai database cho use case

|                                      | Chi tiết                                                                                                                                                                                                          |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Query chậm dần khi data lớn. Index không hiệu quả. Storage tăng nhanh bất thường.                                                                                                                        |
| **Tác động**                | Latency tăng, user experience tệ. Chi phí infra tăng 3-5x. Re-architecture tốn kém.                                                                                                                          |
| **Best Practice**              | **PostgreSQL**làm default choice.**TimescaleDB**cho time-series.**Redis**cho caching/sessions.**MongoDB**cho flexible schema. Đánh giá read/write ratio, data model trước khi chọn. |

### I.3 — Không có API Gateway, client gọi trực tiếp từng service

|                                      | Chi tiết                                                                                                                                                           |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Client phải biết địa chỉ của nhiều services. Authentication logic bị duplicate. Không có rate limiting thống nhất.                                      |
| **Tác động**                | Security gap nghiêm trọng. Khó quản lý và monitor. Breaking changes ảnh hưởng trực tiếp client.                                                          |
| **Best Practice**              | Dùng**API Gateway**(Kong, AWS API Gateway) làm single entry point. Centralized authentication, rate limiting, request routing. API versioning thống nhất. |

### I.4 — Thiếu capacity planning, không đánh giá scalability từ đầu

|                                      | Chi tiết                                                                                                                                            |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Hệ thống chậm vào giờ cao điểm. CPU/Memory thường xuyên > 80%. Database connection pool cạn kiệt.                                        |
| **Tác động**                | Service degradation hoặc downtime vào peak hours. Mất khách hàng. Emergency scaling tốn kém gấp 3-5x.                                        |
| **Best Practice**              | Ước tính DAU và peak concurrency. Tính storage growth rate 12 tháng. Plan**3x headroom**trên expected peak. Review capacity hàng quý. |

## II. RELIABILITY & RESILIENCE

### II.1 — Không có Retry logic với Exponential Backoff

|                                      | Chi tiết                                                                                                                                                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Request thất bại ngay khi gặp transient error. Log đầy lỗi timeout/connection refused nhưng chỉ xảy ra ngắn.                                                                                     |
| **Tác động**                | User thấy lỗi không cần thiết. Tỷ lệ lỗi cao hơn thực tế. Mất trust của user.                                                                                                                 |
| **Best Practice**              | Implement retry với**exponential backoff + jitter**(1s, 2s, 4s...). Giới hạn max 3 lần retry. Chỉ retry transient errors (5xx, timeout). Dùng thư viện:`tenacity`(Python),`polly`(.NET). |

### II.2 — Không có Circuit Breaker, để cascading failures lan tỏa

|                                      | Chi tiết                                                                                                                                                                                                                       |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | 1 service down kéo theo nhiều service khác down. Request bị treo chờ timeout. Connection pool cạn kiệt trên nhiều services.                                                                                            |
| **Tác động**                | **Cascading failure**— toàn bộ hệ thống sập chỉ vì 1 dependency down. Recovery time rất lâu (hàng giờ).                                                                                                       |
| **Best Practice**              | Implement Circuit Breaker với 3 states:**Closed/Open/Half-Open** . Config:`fail_max=5`,`reset_timeout=60s`. Return cached data hoặc default value khi circuit open. Dùng `pybreaker`,`resilience4j`,`Polly`. |

### II.3 — Không đặt Timeout cho external calls

|                                      | Chi tiết                                                                                                                                                                                |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Request bị treo vô thời hạn. Worker processes bị occupied. Hệ thống chậm dần rồi không phản hồi.                                                                            |
| **Tác động**                | **Resource exhaustion**— hết connection, hết thread. Hệ thống ngưng phản hồi hoàn toàn. Cần restart để phục hồi.                                                    |
| **Best Practice**              | LUÔN đặt timeout cho mọi external call (DB, API, cache).**Connect timeout: 3-5s** .**Read timeout: 5-30s**tùy use case. Dùng deadline propagation trong microservices. |

### II.4 — Không có Fallback strategy khi dependency fail

|                                      | Chi tiết                                                                                                                                                                                                           |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Khi 1 service/API down, feature liên quan mất hoàn toàn. User nhận lỗi 500 thay vì degraded experience.                                                                                                      |
| **Tác động**                | User experience bị phá hủy hoàn toàn. Mất revenue từ feature bị ảnh hưởng. User chuyển sang đối thủ.                                                                                                 |
| **Best Practice**              | Thiết kế**Graceful Degradation** : trả về cached data, default value, hoặc simplified response. Ưu tiên: Cache > Default > Error message có ý nghĩa. Feature flags để tắt feature có vấn đề. |

### II.5 — Không có Bulkhead Pattern, không có failure isolation

|                                      | Chi tiết                                                                                                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | 1 endpoint bị overload làm ảnh hưởng toàn bộ API. Slow queries ảnh hưởng các request khác.                                                     |
| **Tác động**                | Không có fault isolation. 1 vấn đề nhỏ có thể làm sập toàn bộ hệ thống.                                                                      |
| **Best Practice**              | Tách connection pools theo service/dependency. Thread pool isolation cho critical paths. Rate limiting per-endpoint. Kubernetes: resource limits per pod. |

### II.6 — Backup không được test định kỳ, DR plan chỉ có trên giấy

|                                      | Chi tiết                                                                                                                                                                                                         |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Chưa bao giờ restore thử từ backup. DR plan viết từ lâu chưa cập nhật. RTO/RPO không ai biết là bao nhiêu.                                                                                          |
| **Tác động**                | Khi sự cố thật sự: backup có thể bị hỏng, restore thất bại, mất data. RTO thực tế gấp 5-10x dự kiến.                                                                                              |
| **Best Practice**              | Test restore backup**hàng tháng** . Chạy DR drill **hàng quý** . Định nghĩa RTO/RPO rõ ràng cho từng service. Backup theo quy tắc **3-2-1** : 3 bản sao, 2 loại media, 1 offsite. |

## III. OBSERVABILITY & MONITORING

### III.1 — Không có monitoring từ đầu, chỉ setup khi có sự cố

|                                      | Chi tiết                                                                                                                                                                       |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Không biết hệ thống đang có vấn đề cho đến khi user báo. Debug bằng cách đọc log thủ công. Không có metrics nào được track.                             |
| **Tác động**                | MTTR (Mean Time To Resolve) rất cao. Không phát hiện vấn đề sớm. Reactive thay vì proactive.                                                                           |
| **Best Practice**              | Setup monitoring**TRƯỚC**khi go live. Track **4 Golden Signals** : Latency, Traffic, Errors, Saturation. Dùng Prometheus + Grafana. Đặt alert cho mỗi signal. |

### III.2 — Alert Fatigue — quá nhiều alert không quan trọng

|                                      | Chi tiết                                                                                                                                                                                                       |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Hàng trăm alerts mỗi ngày. Team bắt đầu ignore alerts. Alert channel bị mute.                                                                                                                           |
| **Tác động**                | Alert thật sự quan trọng bị bỏ lỡ. Incident không được xử lý kịp thời. On-call burnout.                                                                                                           |
| **Best Practice**              | Chỉ alert cho**symptom**(error rate cao) không phải cause (CPU cao). Phân loại severity: P1 (pager), P2 (Slack), P3 (ticket). Review và clean up alerts hàng tháng. Mỗi alert phải có runbook. |

### III.3 — Log không có structure, không tập trung

|                                      | Chi tiết                                                                                                                                                                                   |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Log nằm rải rác trên nhiều server. Không thể search/filter hiệu quả. Thiếu context (request_id, user_id).                                                                         |
| **Tác động**                | Debug mất hàng giờ thay vì phút. Không thể correlate events across services. Không thể tạo alert từ logs.                                                                        |
| **Best Practice**              | **Structured logging**(JSON format).**Centralized log**(ELK Stack, Loki).**Correlation ID**(request_id) qua mọi service. Log levels đúng: DEBUG/INFO/WARN/ERROR/FATAL. |

### III.4 — Không có Distributed Tracing

|                                      | Chi tiết                                                                                                                                                                                              |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Không biết request đi qua những service nào. Không xác định được bottleneck ở đâu. Debug cross-service mất rất nhiều thời gian.                                                     |
| **Tác động**                | Incident resolution chậm gấp 5-10x. Performance issues khó xác định root cause. Không có visibility vào hệ thống.                                                                           |
| **Best Practice**              | Implement**OpenTelemetry**cho tracing. Propagate trace context qua mọi service call. Dùng**Jaeger/Zipkin**để visualize traces. Sample rate: 100% cho errors, 1-10% cho normal traffic. |

### III.5 — Dashboard quá nhiều metrics, không ai hiểu

|                                      | Chi tiết                                                                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Dashboard có 50+ panels. Không ai biết panel nào quan trọng. Mỗi lần có sự cố vẫn phải hỏi senior.                                                                                                                                |
| **Tác động**                | Incident response chậm. Thông tin quan trọng bị chìm. Decision making kém.                                                                                                                                                               |
| **Best Practice**              | Tối đa**7-10 panels**trên 1 dashboard. Tách dashboard theo layer: Business > Service > Infrastructure.**RED method**cho services (Rate, Errors, Duration).**USE method**cho resources (Utilization, Saturation, Errors). |

## IV. DEPLOYMENT & CI/CD

### IV.1 — Deploy thủ công (SSH vào server, pull code, restart)

|                                      | Chi tiết                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Deploy mất hàng giờ. Quy trình khác nhau tùy người. Config sai giữa các lần deploy.                                                                                                 |
| **Tác động**                | Human error cao. Không thể rollback nhanh. Downtime mỗi khi deploy. Không scale được team.                                                                                              |
| **Best Practice**              | CI/CD pipeline tự động:**build > test > security scan > deploy > monitor** . Pipeline as Code (Jenkinsfile, GitHub Actions). Zero-touch deployment — không ai cần SSH vào server. |

### IV.2 — Deploy tất cả traffic cùng lúc (Big Bang deployment)

|                                      | Chi tiết                                                                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Mỗi khi deploy là hồi hộp. Lỗi ảnh hưởng 100% users ngay lập tức. Rollback là phương án cuối cùng.                                                                                                                                |
| **Tác động**                | 100% users bị ảnh hưởng nếu có lỗi. Revenue loss lớn. Reputation damage.                                                                                                                                                                  |
| **Best Practice**              | **Canary Deployment** : deploy cho 1-5% users trước, monitor, rồi tăng dần. **Blue-Green** : 2 environments, switch traffic.**Rolling Deployment**cho Kubernetes.**Feature Flags**để tách deployment và release. |

### IV.3 — Không có rollback strategy hoặc rollback không được test

|                                      | Chi tiết                                                                                                                                                                                       |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Khi có lỗi, team mất hàng giờ để fix forward. Không ai biết cách rollback. Database migration không có reverse.                                                                     |
| **Tác động**                | Downtime kéo dài. Panic trong team. Có thể làm vấn đề tồi tệ hơn.                                                                                                                    |
| **Best Practice**              | **Automated rollback**trigger khi error rate vượt threshold. Rollback được test thường xuyên. Database migration luôn có rollback script. Giữ 3 version trước để rollback. |

### IV.4 — Environment Drift — Staging khác Production

|                                      | Chi tiết                                                                                                                                                                             |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Code chạy tốt trên Staging nhưng lỗi trên Prod. Config, data, infra khác nhau giữa các môi trường.                                                                        |
| **Tác động**                | Bugs chỉ xuất hiện trên Production. Confidence khi deploy thấp. Testing không có giá trị.                                                                                    |
| **Best Practice**              | **Infrastructure as Code**(Terraform/Pulumi). Docker để đảm bảo consistency. Staging mirror Production config (sanitized data). **12-Factor App** : Dev/Prod Parity. |

## V. SECURITY

### V.1 — Hardcode secrets trong source code (API keys, passwords, tokens)

|                                      | Chi tiết                                                                                                                                                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Secrets nằm trong .env file được commit lên Git. Grep source code thấy credentials. Secrets giống nhau giữa dev và prod.                                                                          |
| **Tác động**                | Data breach nghiêm trọng. Compliance violation (GDPR, PCI-DSS). Bị hacker khai thác từ leaked secrets trên GitHub.                                                                                   |
| **Best Practice**              | **Secrets Manager**(HashiCorp Vault, AWS Secrets Manager). Secrets inject lúc runtime qua env vars. Secrets rotation tự động. Git hooks để ngăn commit secrets (`gitleaks`,`trufflehog`). |

### V.2 — Không có input validation, dễ bị SQL Injection/XSS

|                                      | Chi tiết                                                                                                                                                                                 |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | User input được đưa thẳng vào SQL query. Không có sanitization cho HTML output. Error messages lộ database structure.                                                           |
| **Tác động**                | **SQL Injection** : mất toàn bộ database. **XSS** : đánh cắp session của user. Data breach, legal liability.                                                           |
| **Best Practice**              | **Parameterized queries / ORM**(KHÔNG BAO GIỜ string concatenation). Input validation ở cả client và server. Output encoding để chống XSS. Content Security Policy headers. |

### V.3 — Authentication/Authorization yếu hoặc thiếu

|                                      | Chi tiết                                                                                                                                                                                        |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Không có rate limiting cho login. JWT không có expiry hoặc expiry quá dài. RBAC không được enforce nhất quán.                                                                       |
| **Tác động**                | Brute force attacks thành công. Unauthorized access vào sensitive data. Privilege escalation.                                                                                                 |
| **Best Practice**              | **OAuth 2.0 / OpenID Connect** . JWT với short expiry (15-30 phút) + refresh token. RBAC enforce ở API Gateway level.**MFA**cho admin accounts. Rate limiting cho auth endpoints. |

### V.4 — Không có Security Scanning trong CI/CD pipeline

|                                      | Chi tiết                                                                                                                                                                                     |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Dependencies có vulnerabilities đã biết. Code có security flaws không ai biết. Secrets bị commit mà không ai phát hiện.                                                           |
| **Tác động**                | Supply chain attacks. Known vulnerabilities bị exploit. Zero-day exposure cao.                                                                                                               |
| **Best Practice**              | **SAST**(SonarQube) trong CI pipeline.**Dependency scanning**(Snyk, Dependabot).**DAST**cho staging environment.**Secrets detection**(gitleaks) làm pre-commit hook. |

### V.5 — Thiếu Network Security (VPC, Security Groups, WAF)

|                                      | Chi tiết                                                                                                                                                                                                                             |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Database accessible từ Internet. Không có firewall rules. Không có WAF trước web app.                                                                                                                                          |
| **Tác động**                | DDoS attacks làm sập hệ thống. Data exfiltration từ database. Unauthorized access vào internal services.                                                                                                                        |
| **Best Practice**              | **VPC**với private subnets cho DB/internal services.**Security Groups**với least-privilege rules.**WAF**trước public-facing services. DDoS protection (CloudFlare, AWS Shield). Rate limiting ở nhiều layers. |

## VI. TESTING

### VI.1 — Không có automated tests, chỉ manual testing

|                                      | Chi tiết                                                                                                                                                                                 |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Mỗi lần thay đổi code phải test thủ công. Regression bugs xuất hiện liên tục. Test mất nhiều ngày.                                                                          |
| **Tác động**                | Release cycle chậm (tuần -> tháng). Regression bugs nhiều. Confidence khi deploy thấp. Team mệt mỏi.                                                                               |
| **Best Practice**              | **Testing Pyramid** : 70% Unit, 20% Integration, 10% E2E. Automated tests chạy trong CI pipeline. Test coverage target: >= 80% cho critical paths. TDD cho complex business logic. |

### VI.2 — Không có Load/Performance Testing

|                                      | Chi tiết                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Không biết hệ thống chịu được bao nhiêu users. Performance issues chỉ phát hiện khi traffic cao. Không có baseline performance.                                                  |
| **Tác động**                | Hệ thống sập khi có traffic spike (marketing campaign, viral event). Latency tăng, users bỏ đi. Emergency scaling chi phí cao.                                                         |
| **Best Practice**              | Load test trước mỗi major release. Stress test để tìm breaking point.**k6, Locust, JMeter**cho performance testing. Đặt performance budget: **p95 < 200ms, p99 < 500ms** . |

### VI.3 — Test data dùng production data (có PII) hoặc test data quá đơn giản

|                                      | Chi tiết                                                                                                                                                                                                                |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Test database có real user emails, phone numbers. Test không cover edge cases vì data quá clean. Compliance violation risk.                                                                                          |
| **Tác động**                | GDPR/CCPA violation — phạt lên đến 4% doanh thu. Data breach từ test environment. Tests không phát hiện bugs thật sự.                                                                                         |
| **Best Practice**              | **Factory pattern**cho test data (Faker, FactoryBot). Sanitize production data trước khi dùng cho test. Test data phải cover edge cases: unicode, empty strings, large values. Cleanup test data sau mỗi run. |

### VI.4 — Test Coverage chỉ là con số, không đo lường chất lượng test

|                                      | Chi tiết                                                                                                                                                                           |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Coverage 90% nhưng vẫn có nhiều bugs. Tests chỉ check happy path. Mutation testing score thấp.                                                                                |
| **Tác động**                | False confidence từ coverage cao. Critical bugs vẫn lọt qua. Tests không catch regressions.                                                                                     |
| **Best Practice**              | Focus coverage cho critical business logic. Test cả happy path, sad path, edge cases.**Mutation testing**để đo chất lượng test. Review test quality trong code review. |

## VII. ERROR HANDLING

### VII.1 — Silent failures — nuốt exception không log/alert

|                                      | Chi tiết                                                                                                                                                                 |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | `try/catch`với empty catch block. Errors không xuất hiện trong log. Data sai mà không ai biết.                                                                   |
| **Tác động**                | Vấn đề tích tụ theo thời gian. Data corruption không phát hiện. Khi bùng nổ thì đã quá muộn để fix.                                                     |
| **Best Practice**              | **KHÔNG BAO GIỜ**empty catch block. Log mọi exception với đầy đủ context. Phân loại errors: recoverable vs unrecoverable. Alert cho unrecoverable errors. |

### VII.2 — Error messages lộ thông tin hệ thống (stack trace, DB schema)

|                                      | Chi tiết                                                                                                                                                                                                    |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | User thấy stack trace khi có lỗi. Error response chứa database table names. Internal service URLs lộ ra ngoài.                                                                                         |
| **Tác động**                | Information disclosure cho attackers. User experience tệ. Security vulnerability.                                                                                                                           |
| **Best Practice**              | Generic error message cho user ("Something went wrong"). Chi tiết error chỉ trong internal logs. Structured error response:`{code, message, request_id}`. Error tracking với **Sentry/Bugsnag** . |

### VII.3 — Không phân biệt error types, xử lý tất cả như nhau

|                                      | Chi tiết                                                                                                                                                                                    |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Retry cho mọi loại error (cả 400 Bad Request). Không có error classification. Log level giống nhau cho mọi error.                                                                     |
| **Tác động**                | Retry storm cho non-retryable errors. Resource waste. Khó debug vì không phân loại.                                                                                                     |
| **Best Practice**              | Phân loại:**Transient**(retry),**Client**(return 4xx),**Server**(log + alert + retry). Custom exception classes cho từng loại. Error codes có hệ thống để lookup. |

## VIII. DATABASE & DATA MANAGEMENT

### VIII.1 — Thiếu hoặc sai database indexes

|                                      | Chi tiết                                                                                                                                                                    |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Query chậm dần khi data lớn. Full table scan trong slow query log. CPU database cao bất thường.                                                                        |
| **Tác động**                | Latency tăng từ ms lên seconds. Database bị overload. User experience suy giảm nghiêm trọng.                                                                          |
| **Best Practice**              | Index cho mọi WHERE, JOIN, ORDER BY column. Composite index cho multi-column queries.**EXPLAIN ANALYZE**để verify query plan. Monitor slow queries thường xuyên. |

### VIII.2 — N+1 Query Problem

|                                      | Chi tiết                                                                                                                                                                                           |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | 1 trang load phát sinh 100+ DB queries. Log thấy lặp lại cùng 1 query nhiều lần. Response time tăng theo số lượng items.                                                                 |
| **Tác động**                | Latency tăng tuyến tính với data. Database bị overload. Page load > 5s.                                                                                                                        |
| **Best Practice**              | **Eager loading / JOIN**thay vì lazy loading. Batch queries (`WHERE id IN (...)`). DataLoader pattern cho GraphQL. ORM:`select_related`/`prefetch_related`(Django),`include`(Rails). |

### VIII.3 — Connection Pool không được config đúng

|                                      | Chi tiết                                                                                                                                                                                              |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | "Too many connections" errors. Connections không được release. New connections mất nhiều thời gian.                                                                                             |
| **Tác động**                | Database unreachable. Toàn bộ hệ thống bị ảnh hưởng. Cascading failures.                                                                                                                       |
| **Best Practice**              | Pool size =**(CPU cores * 2) + number_of_disks** . Max pool size phù hợp với DB max_connections. Connection timeout và idle timeout. Health check cho connections. Monitor pool utilization. |

### VIII.4 — Database migration không có rollback, chạy trực tiếp trên prod

|                                      | Chi tiết                                                                                                                                                                                          |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Migration thất bại để database ở trạng thái không nhất quán. Không có cách quay lại. Schema và code không đồng bộ.                                                              |
| **Tác động**                | Data corruption. Downtime kéo dài. Mất data.                                                                                                                                                    |
| **Best Practice**              | Mỗi migration có**UP và DOWN**script. Test migration trên staging trước. Blue-Green migration cho zero-downtime. **Backup TRƯỚC khi migrate** . Flyway/Alembic/Prisma Migrate. |

## IX. API DESIGN

### IX.1 — Không có API versioning

|                                      | Chi tiết                                                                                                                                                                           |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Thay đổi API làm client bị lỗi. Không thể update API mà không break backward compatibility. Client và server phải deploy cùng lúc.                                     |
| **Tác động**                | Breaking changes ảnh hưởng mọi client. Không thể evolve API. Tight coupling giữa client và server.                                                                          |
| **Best Practice**              | **URL versioning**(`/v1/`,`/v2/`) hoặc Header versioning. Deprecation policy rõ ràng (3-6 tháng). Backward compatibility mặc định. API changelog cho mỗi version. |

### IX.2 — Không có Pagination, trả về tất cả data

|                                      | Chi tiết                                                                                                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | API trả về 10,000+ records trong 1 response. Response size rất lớn (MB). Client bị chậm/crash khi load.                                              |
| **Tác động**                | Memory overflow ở cả server và client. Network bandwidth lãng phí. Terrible UX.                                                                       |
| **Best Practice**              | **Cursor-based pagination**cho large datasets. Default page size: 20-50. Max page size: 100-200. Include total count và next cursor trong response. |

### IX.3 — Không có Rate Limiting

|                                      | Chi tiết                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | 1 client có thể gửi vô hạn requests. Không có bảo vệ trước API abuse. Resource không được chia sẻ công bằng.                                                                    |
| **Tác động**                | DDoS dễ dàng. 1 client có thể làm sập hệ thống. Chi phí infra tăng không kiểm soát.                                                                                                  |
| **Best Practice**              | Rate limiting per client/API key. Multiple tiers: 100/min cho free, 1000/min cho paid. Return**429 Too Many Requests**với Retry-After header. Token bucket hoặc sliding window algorithm. |

## X. CONFIGURATION & SECRETS MANAGEMENT

### X.1 — Configuration Drift — config khác nhau giữa environments mà không ai biết

|                                      | Chi tiết                                                                                                                                                                           |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | "It works on my machine" là câu nói thường xuyên. Staging pass nhưng Prod fail. Không có single source of truth cho config.                                                |
| **Tác động**                | Bugs chỉ xuất hiện trên Production. Debug mất thời gian. Confidence thấp.                                                                                                    |
| **Best Practice**              | **Infrastructure as Code**(Terraform). Config tách khỏi code, inject qua env vars. Centralized config service (Consul, AWS Parameter Store). Config validation khi startup. |

### X.2 — Secrets trong config files, .env files commit lên Git

|                                      | Chi tiết                                                                                                                                                                               |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | .env file trong Git repo. Secrets giống nhau giữa environments. Không có secrets rotation.                                                                                          |
| **Tác động**                | Secrets bị leak. Không thể revoke mà không deploy lại. Compliance violation.                                                                                                      |
| **Best Practice**              | **Secrets Manager**(Vault, AWS SM). .env**KHÔNG BAO GIỜ**commit lên Git (.gitignore). Secrets rotation tự động (30-90 ngày). Audit log cho mỗi lần access secrets. |

### X.3 — Không có config validation, ứng dụng start với config sai

|                                      | Chi tiết                                                                                                                                                                                |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Ứng dụng start thành công nhưng fail khi xử lý request đầu tiên. Config sai chỉ bị phát hiện khi có traffic.                                                              |
| **Tác động**                | Delayed failure — khó debug. Partial failures. Downtime sau deploy.                                                                                                                    |
| **Best Practice**              | Validate tất cả config khi startup (**fail fast** ). Schema validation cho config files. Health check endpoint kiểm tra config. Config changes thông qua CI/CD, không manual. |

## XI. CODE QUALITY & MAINTAINABILITY

### XI.1 — Cyclomatic Complexity cao, function quá dài (>50 dòng)

|                                      | Chi tiết                                                                                                                                                       |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Functions có nhiều nested if/else. Khó đọc, khó hiểu code. Bug fix tạo ra bugs mới.                                                                    |
| **Tác động**                | Maintenance cost tăng theo thời gian. Onboarding developer mới mất nhiều thời gian. Technical debt tích tụ.                                             |
| **Best Practice**              | Max**20 cyclomatic complexity**per function. Single Responsibility Principle. Extract method refactoring.**SonarQube**để đo lường và enforce. |

### XI.2 — Không có Code Review process

|                                      | Chi tiết                                                                                                                                                                                                          |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Code được merge trực tiếp vào main. Bugs chỉ được phát hiện bởi users. Không có knowledge sharing.                                                                                                  |
| **Tác động**                | Bug rate cao. Code quality giảm. Team không học từ nhau. Security vulnerabilities lọt qua.                                                                                                                    |
| **Best Practice**              | Bắt buộc**Pull Request**với >= 1 reviewer. Code review checklist: functionality, readability, performance, security, tests. Automated checks (linter, formatter) trước review. Small PRs (< 400 lines). |

### XI.3 — Technical Debt không được track và giải quyết

|                                      | Chi tiết                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | TODO comments khắp nơi trong code. Workarounds chồng chéo. Refactoring bị delay liên tục.                                                                                                  |
| **Tác động**                | Velocity giảm 20-40% mỗi năm. Developer frustration tăng. Innovation bị cản trở.                                                                                                           |
| **Best Practice**              | Track tech debt trong backlog với priority. Dành**15-20% sprint capacity**cho tech debt. Refactor incrementally, không big-bang rewrite. Measure: code churn, defect density, lead time. |

## XII. INFRASTRUCTURE & CONTAINERIZATION

### XII.1 — Docker images quá lớn, không optimize

|                                      | Chi tiết                                                                                                                                                           |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Image size > 1GB. Build time > 10 phút. Pull image mất nhiều thời gian.                                                                                         |
| **Tác động**                | Deploy chậm. Storage cost cao. Scale-out chậm vì pull image lâu.                                                                                                |
| **Best Practice**              | **Multi-stage builds** . Alpine/Distroless base images.`.dockerignore`file. Layer caching: dependencies trước, code sau. Target image **< 100MB** . |

### XII.2 — Container chạy với root privileges

|                                      | Chi tiết                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Dockerfile không có USER instruction. Container có thể access host filesystem. Security scan báo privilege escalation risk.                                                               |
| **Tác động**                | Container escape attack. Host compromise. Data breach toàn hệ thống.                                                                                                                        |
| **Best Practice**              | **USER non-root**trong Dockerfile. Read-only filesystem khi có thể. Security context trong K8s:`runAsNonRoot`,`readOnlyRootFilesystem`. Image scanning với **Trivy/Snyk** . |

### XII.3 — Kubernetes không có Resource Limits

|                                      | Chi tiết                                                                                                                                                                            |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | 1 pod chiếm hết resources của node. Noisy neighbor problem. OOMKilled events thường xuyên.                                                                                     |
| **Tác động**                | Không có resource isolation. Các pod khác bị ảnh hưởng. Node instability.                                                                                                    |
| **Best Practice**              | **Requests và Limits**cho mọi container. Requests = typical usage, Limits = max allowed. LimitRange và ResourceQuota cho namespace. Monitor resource utilization để tune. |

### XII.4 — Liveness/Readiness Probe không đúng hoặc thiếu

|                                      | Chi tiết                                                                                                                                                                                                                                                                 |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Pod bị kill liên tục (CrashLoopBackOff). Traffic được route đến pod chưa sẵn sàng. Health check không phản ánh trạng thái thật.                                                                                                                          |
| **Tác động**                | Service disruption. Request failures. Pod restart loops.                                                                                                                                                                                                                  |
| **Best Practice**              | **Readiness probe** : kiểm tra app sẵn sàng nhận traffic (check DB, cache). **Liveness probe** : kiểm tra app còn sống (đơn giản, nhanh).**Startup probe**cho apps khởi động chậm. Tune timing:`initialDelaySeconds`,`periodSeconds`. |

## XIII. DOCUMENTATION & RUNBOOKS

### XIII.1 — Documentation outdated hoặc không tồn tại

|                                      | Chi tiết                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | README viết từ 2 năm trước. Architecture diagram không khớp với thực tế. New member mất 2-3 tuần để onboard.                                                                     |
| **Tác động**                | Onboarding chậm. Knowledge silo. Sai lầm khi không hiểu hệ thống.                                                                                                                        |
| **Best Practice**              | **Docs-as-Code** : documentation sống cùng source code.**Architecture Decision Records**(ADRs). Auto-generate API docs từ code (Swagger/OpenAPI). Review docs trong PR process. |

### XIII.2 — Không có Runbooks/Playbooks cho incident response

|                                      | Chi tiết                                                                                                                                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Triệu chứng / Dấu hiệu** | Mỗi khi có sự cố, team làm từng bước từ đầu. On-call không biết bắt đầu từ đâu. Resolution time phụ thuộc vào ai đang trực.                                        |
| **Tác động**                | MTTR cao và không ổn định. Stress cho on-call. Có thể làm vấn đề tồi tệ hơn do panic.                                                                                        |
| **Best Practice**              | Runbook cho mỗi alert: symptom, diagnosis steps, resolution steps, escalation path. Template chuẩn cho mỗi runbook. Review và update sau mỗi incident. Liên kết runbook với alert. |

### XIII.3 — API documentation thiếu hoặc không chính xác

|                                      | Chi tiết                                                                                                                                                              |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Client developers phải đọc source code để hiểu API. API behavior khác với document. Không có examples.                                                       |
| **Tác động**                | Integration chậm và nhiều lỗi. Developer experience tệ. Client team frustrated.                                                                                   |
| **Best Practice**              | **OpenAPI/Swagger**specification. Auto-generate từ code annotations. Examples cho mỗi endpoint. Changelog cho mỗi thay đổi. Postman collection cho testing. |

## XIV. PRODUCTION READINESS

### XIV.1 — Không có Production Readiness Review (PRR) trước khi go live

|                                      | Chi tiết                                                                                                                                                                       |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Launch rồi mới phát hiện thiếu monitoring. Không có on-call schedule. Không có SLO/SLA.                                                                                |
| **Tác động**                | Incident đầu tiên không được xử lý tốt. User experience tệ ngay từ đầu. Reputation damage sớm.                                                                   |
| **Best Practice**              | **PRR Checklist**bắt buộc trước go-live. Cover: monitoring, alerting, on-call, runbooks, capacity, security, backup, rollback. Sign-off từ Engineering Lead và SRE. |

### XIV.2 — SLO/SLI/SLA không được định nghĩa hoặc không realistic

|                                      | Chi tiết                                                                                                                                                                                                                                                                                      |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Không biết "hệ thống ổn định" nghĩa là gì. Không có ngưỡng để quyết định rollback. Không có error budget.                                                                                                                                                                 |
| **Tác động**                | Không có cơ sở để đánh giá chất lượng. Không cân bằng được reliability vs velocity. Arguments giữa dev và ops.                                                                                                                                                             |
| **Best Practice**              | Định nghĩa**SLIs**(chỉ số đo): latency p99, error rate, availability. Đặt**SLOs**(mục tiêu): 99.9% availability, p99 < 200ms.**SLA**= SLO + hậu quả (thưởng/phạt).**Error budget**= 100% - SLO để quyết định invest vào reliability hay features. |

### XIV.3 — Không có Incident Management Process

|                                      | Chi tiết                                                                                                                                                                                                                         |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Khi có sự cố, không ai biết ai chịu trách nhiệm. Communication hỗn loạn. Post-mortem không được làm.                                                                                                               |
| **Tác động**                | Resolution chậm. Lỗi lặp lại. Team stress và demotivated.                                                                                                                                                                    |
| **Best Practice**              | **Incident Commander**role rõ ràng. Communication channel (Slack #incident). Severity levels (P1-P4) với response time.**Blameless post-mortem**sau mỗi P1/P2. Action items được track đến khi hoàn thành. |

## XV. MLOps & AI SYSTEMS

### XV.1 — Không monitor Model Drift — model suy giảm theo thời gian

|                                      | Chi tiết                                                                                                                                                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Model accuracy giảm dần mà không ai biết. Predictions ngày càng sai. User feedback xấu đi.                                                                                                        |
| **Tác động**                | Business decisions dựa trên dự đoán sai. Revenue giảm. User mất niềm tin vào AI features.                                                                                                         |
| **Best Practice**              | Monitor**data drift**(PSI, KS-test) và **concept drift** . A/B testing cho model updates. Shadow deployment trước khi thay thế. Retraining pipeline tự động khi drift vượt threshold. |

### XV.2 — Model Serving không có optimization, latency cao

|                                      | Chi tiết                                                                                                                                                                                            |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Inference time > 1s. GPU utilization thấp. Không có batching.                                                                                                                                     |
| **Tác động**                | User experience tệ. Cost/request cao. Không scale được.                                                                                                                                         |
| **Best Practice**              | Model optimization:**quantization, pruning, distillation** . Batching inference requests. Caching cho frequent predictions. GPU sharing/Multi-model serving. **TensorRT/ONNX Runtime** . |

### XV.3 — Model Versioning hỗn loạn, không biết model nào đang chạy trên prod

|                                      | Chi tiết                                                                                                                                                                                                                  |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Triệu chứng / Dấu hiệu** | Không biết model được train từ data nào. Không thể reproduce results. Rollback model là nightmare.                                                                                                               |
| **Tác động**                | Compliance risk (không explain được model). Không thể debug model issues. Không thể rollback.                                                                                                                      |
| **Best Practice**              | **Model Registry**(MLflow, Weights & Biases). Version: model + data + hyperparams + metrics. Staged deployment: staging > canary > production. **Lineage tracking** : data -> training -> model -> deployment. |

## TỔNG KẾT & HƯỚNG DẪN ƯU TIÊN

### Thứ tự ưu tiên triển khai

| Ưu tiên                  | Domain                                      | Timeline   |
| -------------------------- | ------------------------------------------- | ---------- |
| **P1 - Critical**    | Security, Reliability, Error Handling       | Tuần 1-2  |
| **P2 - High**        | Observability, Deployment, Database         | Tuần 3-4  |

### Nguyên tắc vàng

1. **Bắt đầu từ những thứ đang gây đau (pain points) nhất cho team.**

3. **Mỗi sprint dành 15-20% capacity** cho việc cải thiện production quality.
4. **Review checklist này mỗi quý** để cập nhật và đánh giá lại.
5. **Production Quality là hành trình liên tục, không phải đích đến.**
