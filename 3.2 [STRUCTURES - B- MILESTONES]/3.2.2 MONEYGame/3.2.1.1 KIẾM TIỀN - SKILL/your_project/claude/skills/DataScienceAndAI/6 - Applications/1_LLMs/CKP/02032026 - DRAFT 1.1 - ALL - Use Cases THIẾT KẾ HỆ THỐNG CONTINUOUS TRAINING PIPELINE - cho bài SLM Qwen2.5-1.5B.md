Orchestrated comprehensive fine-tuning guide for Qwen model

Orchestrated comprehensive fine-tuning guide for Qwen model

OK, mình thấy bạn đang chạy **Qwen2.5-1.5B-Instruct-AWQ** trên vLLM cho emotion service. Mình sẽ hướng dẫn flow fine-tune từ A-Z, chưa code gì hết.

---

## Hiểu bức tranh tổng thể trước

Fine-tune SLM gồm 5 giai đoạn: **Data → Format → Train → Eval → Deploy**. Với con Qwen2.5-1.5B thì rất nhẹ, flow khá nhanh.

## 1. Chuẩn bị Data (quan trọng nhất, chiếm 70% effort)

**Bạn cần trả lời mấy câu này trước:**

- Emotion service hiện tại đang detect những emotion gì? (vd: happy, sad, angry, neutral...)
- Input là gì? (text chat của trẻ em học tiếng Anh? audio transcript?)
- Output mong muốn là gì? (label emotion? label + confidence? label + explanation?)
- Model hiện tại đang sai ở đâu? (nhầm emotion nào với emotion nào? không hiểu context tiếng Việt?)

**Format data cho fine-tune:**

Data cần ở dạng instruction-response pairs. Ví dụ với emotion detection:

```
instruction: "Phân tích cảm xúc của câu sau: 'Con không muốn học nữa, chán lắm'"
response: "emotion: sad, confidence: high, reason: Người dùng thể hiện sự chán nản và muốn từ bỏ"
```

Hoặc đơn giản hơn nếu chỉ cần label:

```
instruction: "Classify emotion: 'I love this game so much!'"  
response: "happy"
```

**Số lượng data cần bao nhiêu?**

- Minimum: 500-1000 samples đã cho kết quả khá tốt với LoRA trên SLM 1.5B
- Tốt: 3000-5000 samples
- Nếu có imbalanced classes (vd: neutral nhiều, angry ít) → cần balance lại hoặc oversample class thiểu số

## 2. Chọn phương pháp fine-tune

Với Qwen2.5-1.5B, bạn có 3 lựa chọn:

**LoRA/QLoRA (khuyến nghị)** → chỉ train một phần nhỏ parameters (~1-5%), nhanh, ít VRAM. Đây là lựa chọn tốt nhất cho bài toán classification/emotion như của bạn.

**Full fine-tune** → train toàn bộ model. Tốn VRAM hơn nhưng con 1.5B nhỏ nên vẫn feasible. Chỉ cần khi muốn thay đổi behavior sâu.

**Với emotion detection, LoRA là đủ và thừa.**

## 3. Chọn base model để fine-tune

⚠️ **Điểm quan trọng:** Bạn đang serve bản **AWQ** (quantized), nhưng khi fine-tune bạn phải train từ bản gốc:

- `Qwen/Qwen2.5-1.5B-Instruct` (bản full, cho instruction-following)
- hoặc `Qwen/Qwen2.5-1.5B` (bản base, nếu muốn train từ đầu)

→ Dùng bản **Instruct** vì emotion detection là instruction-following task. Sau khi fine-tune xong, bạn mới quantize lại sang AWQ hoặc GGUF để deploy.

## 4. Tài nguyên cần gì?

- **GPU**: RTX 4060/4070 (8-16GB VRAM) hoặc Colab T4 free là đủ
- **Thời gian**: 20-60 phút cho 1000-3000 samples
- **Tool**: Unsloth (dễ nhất, tối ưu nhất cho Qwen) hoặc Axolotl (nếu muốn MLOps-ready)

## 5. Flow deploy sau fine-tune

```
Train (Unsloth + LoRA)
  → Merge LoRA adapter vào base model
    → Quantize sang AWQ (dùng AutoAWQ)
      → Thay thế model trong docker-compose (đổi path model)
        → vLLM serve như cũ, config gần như không đổi
```

Docker-compose của bạn gần như giữ nguyên, chỉ đổi dòng `--model` từ `Qwen/Qwen2.5-1.5B-Instruct-AWQ` sang path model fine-tuned của bạn (push lên HuggingFace hoặc mount local volume).

## 6. Eval - đo lường kết quả

Trước khi train, bạn cần chuẩn bị **test set** (tách 10-20% data ra không train). Sau đó so sánh:

- Accuracy / F1-score trên emotion labels
- Confusion matrix (xem model nhầm emotion nào với emotion nào)
- So sánh base model vs fine-tuned model trên cùng test set

Charted AI-driven labeling methodology for model fine-tuning

Charted AI-driven labeling methodology for model fine-tuning

Đây là câu hỏi rất đúng vì gán nhãn thủ công thì tốn thời gian kinh khủng. Dùng AI gán nhãn (LLM-as-labeler) là cách phổ biến nhất hiện tại, nhưng phải làm đúng cách thì data mới chất lượng.

---

## Tổng quan: Pipeline gán nhãn bằng AI

```
Raw data (chat logs, text)
  → Bước 1: Thiết kế Label Schema
    → Bước 2: Viết Labeling Prompt cho LLM mạnh (GPT-4o / Claude)
      → Bước 3: Chạy batch labeling
        → Bước 4: Human review sample (quality check)
          → Bước 5: Lọc & fix data kém chất lượng
            → Dataset sạch → Fine-tune
```

## Bước 1: Thiết kế Label Schema thật rõ ràng

Đây là bước nhiều người bỏ qua nhưng quyết định chất lượng toàn bộ dataset.

**Cần định nghĩa cụ thể từng label:**

Không nên chỉ viết "happy, sad, angry". Phải có **definition + examples + edge cases** cho mỗi label. Ví dụ:

```
EMOTION LABELS:

1. happy - Vui vẻ, phấn khích, hài lòng
   ✅ "Yay I got it right!" 
   ✅ "Con làm được rồi!"
   ❌ "OK fine" (đây là neutral, không phải happy)

2. frustrated - Bực bội, khó chịu vì không làm được
   ✅ "I don't understand this at all"
   ✅ "Sao khó quá vậy"
   ❌ "I don't want to study" (đây là disengaged, không phải frustrated)

3. confused - Bối rối, không hiểu
   ✅ "Wait what? I don't get it"
   ✅ "Hả? Cái này là sao?"

4. disengaged - Chán, mất hứng, muốn dừng
   ✅ "Can we stop now?"
   ✅ "Chán lắm rồi"

5. neutral - Bình thường, không cảm xúc rõ ràng
   ✅ "OK", "Yes", "The answer is B"
```

**Tips quan trọng:** Labels phải **mutually exclusive** (không overlap) và **collectively exhaustive** (cover hết các trường hợp). Nếu có câu nào không fit vào label nào → thêm label hoặc thêm "other/uncertain".

## Bước 2: Viết Labeling Prompt (cốt lõi nhất)

Dùng một LLM mạnh (Claude Opus, GPT-4o) làm "labeler". Prompt phải cực kỳ chi tiết:

```
SYSTEM PROMPT cho LLM labeler:
---
Bạn là chuyên gia phân tích cảm xúc trẻ em 6-12 tuổi 
trong context học tiếng Anh với robot AI.

TASK: Phân loại cảm xúc của câu nói dưới đây.

LABELS (chọn đúng 1):
- happy: [định nghĩa + ví dụ]
- frustrated: [định nghĩa + ví dụ]  
- confused: [định nghĩa + ví dụ]
- disengaged: [định nghĩa + ví dụ]
- neutral: [định nghĩa + ví dụ]

QUY TẮC:
1. Xét CONTEXT là trẻ em đang học, không phải người lớn
2. "I don't know" trong context học → confused, 
   KHÔNG phải disengaged
3. Câu ngắn 1-2 từ như "OK", "Yes" → neutral 
   trừ khi có dấu hiệu cảm xúc rõ
4. Nếu không chắc chắn, gán confidence = low
5. Xét cả tiếng Việt lẫn tiếng Anh

OUTPUT FORMAT (JSON only, không giải thích gì thêm):
{
  "emotion": "<label>",
  "confidence": "high|medium|low",
  "reasoning": "<1 câu ngắn giải thích>"
}

INPUT: "{text}"
```

**Tại sao cần field `reasoning`?** Đây là trick quan trọng — khi LLM phải giải thích lý do, nó sẽ gán nhãn chính xác hơn nhiều (Chain-of-Thought effect). Sau này bạn có thể bỏ reasoning đi, chỉ giữ emotion label.

**Tại sao cần field `confidence`?** Để ở bước quality check, bạn ưu tiên review những sample có confidence = low trước.

## Bước 3: Chạy Batch Labeling

**Cách 1: Dùng Claude/GPT API trực tiếp**

- Gọi API với từng sample, parse JSON response
- Cost: ~$5-15 cho 5000 samples với Claude Sonnet hoặc GPT-4o-mini
- GPT-4o-mini hoặc Claude Haiku đủ tốt cho labeling task đơn giản, không cần model đắt tiền

**Cách 2: Multi-LLM voting (best practice cho data chất lượng cao)**

```
Mỗi sample → gọi 2-3 LLM khác nhau gán nhãn
  → Nếu 2/3 hoặc 3/3 đồng ý → giữ label đó (high confidence)
  → Nếu không đồng ý → đánh dấu để human review
```

Ví dụ: dùng Claude Haiku + GPT-4o-mini + Gemini Flash. Cả 3 đều rẻ, chạy nhanh. Nếu 2/3 nói "frustrated" → label = frustrated. Nếu 1 nói frustrated, 1 nói confused, 1 nói sad → flagged để human xem.

**Cách 3: LLM mạnh label → LLM mạnh verify (2-pass)**

```
Pass 1: Claude Sonnet gán nhãn toàn bộ
Pass 2: GPT-4o review lại những sample khó 
        (confidence = low hoặc edge cases)
```

## Bước 4: Human Review (không bỏ qua được)

Dù AI label tốt đến đâu, bạn **phải** human review. Nhưng không cần review hết:

**Chiến lược review thông minh:**

- Review 100% samples có confidence = "low" (thường ~10-15% tổng data)
- Random sample 5-10% từ nhóm confidence = "high" để spot-check
- Review 100% samples mà multi-LLM không đồng ý nhau
- Tính **inter-annotator agreement** giữa AI và human → nếu > 90% thì AI label đủ tốt

**Metric cần theo dõi:**

```
AI accuracy = (số sample AI label đúng) / (số sample human đã review)

Mục tiêu: > 85-90% accuracy
Nếu < 80% → prompt chưa tốt, cần cải thiện prompt rồi chạy lại
```

## Bước 5: Data Cleaning & Final Dataset

Sau khi review xong:

- Loại bỏ samples mà cả AI và human đều không chắc chắn
- Fix labels sai
- Kiểm tra **class distribution** — nếu 80% neutral, 2% angry thì model sẽ bias nặng

**Cách xử lý imbalanced data:**

- Oversample class thiểu số (duplicate hoặc paraphrase)
- Dùng LLM generate thêm synthetic data cho class ít (vd: "Viết 50 câu mà trẻ 8 tuổi nói khi frustrated khi học tiếng Anh")
- Hoặc dùng weighted loss khi train

## Tổng kết Best Practices

|Practice|Tại sao|
|---|---|
|Label schema rõ ràng + ví dụ + edge cases|LLM không đoán mò, consistent hơn|
|Bắt LLM output reasoning|Chain-of-thought → label chính xác hơn|
|Bắt LLM output confidence|Biết sample nào cần human review|
|Multi-LLM voting hoặc 2-pass|Giảm bias của 1 model đơn lẻ|
|Human review 10-20%|Quality gate, tính accuracy|
|Check class distribution|Tránh model bias về class đông|

## Bước tiếp theo cụ thể cho bạn:

1. **Liệt kê ra** danh sách emotion labels bạn cần cho PIKA
2. **Viết definition + examples** cho mỗi label (giống template ở Bước 1)
3. **Export raw data** từ hệ thống hiện tại (chat logs của trẻ)
4. Sau đó mình sẽ giúp bạn viết labeling prompt + batch script

Architected comprehensive automated MLOps pipeline for continuous model training

Architected comprehensive automated MLOps pipeline for continuous model training

Đây là câu hỏi rất hay — bạn đang muốn xây **Continuous Training Pipeline** (CT Pipeline) cho SLM. Đây là bài toán MLOps/LLMOps cấp production. Mình sẽ thiết kế từ tổng quan đến chi tiết.

---

## Bức tranh tổng thể: Closed-Loop LLMOps

```
┌─────────────────────────────────────────────────────────────────┐
│                    CLOSED-LOOP CT PIPELINE                       │
│                                                                  │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 1. DATA  │───▶│ 2. TRAIN │───▶│ 3. EVAL  │───▶│4. DECIDE │  │
│  │ COLLECT  │    │          │    │          │    │ PROMOTE? │  │
│  └──────────┘    └──────────┘    └──────────┘    └────┬─────┘  │
│       ▲                                               │         │
│       │           ┌──────────┐    ┌──────────┐        │         │
│       │           │ 6. MON   │◀───│ 5. DEPLOY│◀───────┘         │
│       │           │ ITOR     │    │ (canary) │   YES             │
│       └───────────┴──────────┘    └──────────┘                  │
│            feedback loop                                         │
└─────────────────────────────────────────────────────────────────┘
```

Gồm 6 giai đoạn, mỗi giai đoạn chạy tự động và trigger giai đoạn tiếp theo.

---

## Giai đoạn 1: Data Collection & Auto-Labeling

**Mục tiêu:** Liên tục thu thập data mới từ production, tự gán nhãn, tự kiểm tra chất lượng.

```
Production traffic (emotion-api)
  │
  ▼
Log collector (ghi lại input + model output + user feedback)
  │
  ▼
Auto-labeler (LLM mạnh gán nhãn, multi-vote)
  │
  ▼
Quality gate (lọc confidence thấp → human review queue)
  │
  ▼
Data store (versioned dataset - DVC hoặc HuggingFace Dataset)
```

**Các nguồn signal quan trọng:**

- **Implicit feedback:** User có tiếp tục tương tác sau khi bot respond theo emotion không? Nếu bot detect "happy" mà user bỏ cuộc → có thể sai
- **Explicit feedback:** Nếu có nút 👍👎 hoặc teacher đánh giá
- **Hard examples:** Những sample mà model hiện tại confidence thấp (logprobs từ vLLM) → đây là gold mine để improve
- **Distribution shift:** Input mới khác pattern cũ (new slang, new topics)

**Auto-labeling pipeline:**

```
Mỗi ngày/tuần chạy batch:
1. Lấy N samples mới từ production logs
2. Gọi Claude/GPT gán nhãn (có reasoning + confidence)
3. Multi-vote nếu budget cho phép
4. Samples confidence HIGH → auto-add vào training pool
5. Samples confidence LOW → đẩy vào human review queue
6. Tính label distribution → alert nếu drift
```

**Trigger chuyển sang giai đoạn 2:**

- Khi accumulated data mới đạt threshold (vd: 500+ samples mới)
- Hoặc theo schedule (weekly/bi-weekly)
- Hoặc khi monitoring phát hiện model degradation

---

## Giai đoạn 2: Auto Training

**Mục tiêu:** Tự động fine-tune khi có đủ data mới, reproducible 100%.

```
Trigger (đủ data / schedule / quality drop)
  │
  ▼
Data preparation
  ├── Merge new data + existing validated data
  ├── Train/Val/Test split (stratified by label)
  ├── Data versioning (tag: v2.3.1)
  │
  ▼
Training job
  ├── Base: luôn từ Qwen2.5-1.5B-Instruct (KHÔNG từ model cũ đã fine-tune)
  ├── Method: LoRA/QLoRA
  ├── Hyperparams: từ config file (versioned)
  ├── Hardware: auto-provision GPU (RunPod / on-prem)
  │
  ▼
Artifacts
  ├── LoRA adapter weights
  ├── Training metrics (loss curve, lr schedule)
  ├── Training config snapshot
  └── Data version reference
```

**Best practices training:**

- **Luôn train từ base model gốc**, không train incremental từ checkpoint cũ. Lý do: tránh catastrophic forgetting tích lũy, dễ reproduce, dễ debug
- **Toàn bộ config là code** — version control trên Git, không manual tweak
- **Mỗi training run có unique ID**, log đầy đủ lên experiment tracker (MLflow, Weights & Biases, hoặc đơn giản là structured logs)

**Anti-pattern cần tránh:**

```
❌ Train tiếp từ model fine-tuned lần trước
   → Catastrophic forgetting tích lũy, model ngày càng tệ

❌ Merge data mới mà không validate 
   → Garbage in, garbage out

❌ Dùng hyperparams cố định mãi
   → Data thay đổi, optimal hyperparams cũng thay đổi
   → Có thể chạy nhẹ hyperparameter search mỗi N lần train
```

---

## Giai đoạn 3: Auto Evaluation (Gate quan trọng nhất)

**Mục tiêu:** Tự động đánh giá model mới, so sánh với model đang chạy production, quyết định có promote hay không.

```
Model mới (candidate)
  │
  ▼
Eval suite chạy tự động
  │
  ├── Benchmark test set (cố định, không bao giờ dùng để train)
  │     ├── Accuracy / F1 / Precision / Recall per class
  │     ├── Confusion matrix
  │     └── Latency (inference speed)
  │
  ├── Regression test set (edge cases đã từng fix)
  │     └── "Những case model cũ từng sai, đã fix"
  │         → Model mới PHẢI pass hết
  │
  ├── Golden test set (human-curated, high quality)
  │     └── 100-200 samples, gán nhãn bởi expert
  │         → Đây là "source of truth"
  │
  ├── A/B comparison với model hiện tại (production)
  │     └── Cùng input → so output
  │
  └── Safety checks
        ├── Không output toxic content
        ├── Không hallucinate emotion
        └── Latency < threshold (cho vLLM serving)
```

**Promotion criteria (ví dụ):**

yaml

````yaml
promotion_rules:
  # Model mới phải tốt hơn model cũ
  accuracy_improvement_min: 0.5%    # ít nhất +0.5%
  f1_improvement_min: 0.3%
  
  # Không được tệ hơn ở bất kỳ class nào
  per_class_regression_max: -2%     # không class nào drop > 2%
  
  # Regression tests
  regression_test_pass_rate: 100%   # phải pass hết
  
  # Performance
  p95_latency_max_ms: 100          # không chậm hơn
  
  # Safety
  safety_check: pass
  
  # Decision
  auto_promote: true                # nếu tất cả pass → tự deploy
  require_human_approval: false     # true nếu muốn thêm 1 bước manual
```

**3 loại test set phải tách biệt — rất quan trọng:**
```
┌─────────────────────────────────────────────────┐
│ Training data     → dùng để train               │
│ Validation data   → dùng để tune hyperparams    │
│ Benchmark test    → CỐ ĐỊNH, đo performance     │
│ Regression test   → TÍCH LŨY edge cases đã fix  │
│ Golden test       → HUMAN-CURATED, source truth  │
└─────────────────────────────────────────────────┘

KHÔNG BAO GIỜ để test data lọt vào training data.
```

**Regression test set đặc biệt quan trọng:** Mỗi khi phát hiện model sai 1 case → fix xong → thêm case đó vào regression test. Set này chỉ tăng, không giảm. Nó đảm bảo model mới không "quên" những gì đã fix.

## Giai đoạn 4: Auto Decision — Promote hay Reject
```
Eval results
  │
  ▼
Decision engine
  │
  ├── ALL criteria pass?
  │     ├── YES → Tag model (v2.3.1), promote to staging
  │     └── NO  → Log reason, alert team, giữ model cũ
  │
  ├── Edge case: model mới tốt hơn overall nhưng 
  │   1 class bị regression?
  │     └── Alert team, KHÔNG auto-promote
  │         (cần human review)
  │
  └── Edge case: model mới ngang bằng model cũ?
        └── KHÔNG promote (không có lý do thay đổi)
```

**Principle: conservative by default** — chỉ promote khi chắc chắn tốt hơn. Khi nghi ngờ → giữ model cũ → alert team.

## Giai đoạn 5: Auto Deploy (Canary/Blue-Green)

**Đây là nơi connect với docker-compose hiện tại của bạn:**

  │
  ▼

  │
  ▼

  │

Canary deployment

  ├── 90% traffic → model cũ  (vllm-qwen-stable)

  ├── Latency tăng?     → auto rollback
  ├── Emotion distribution lệch bất thường? → auto rollback
  ├── Tất cả OK?        → ramp lên 50% → 100%
  │
  ▼
Full rollout → model mới thành stable
  └── Model cũ giữ lại làm rollback backup
```

## Giai đoạn 6: Monitoring & Feedback Loop

**Đây là thứ biến pipeline từ "chạy 1 lần" thành "tự cải thiện liên tục":**
```
Production model đang serve
  │
  ▼
Monitor liên tục
  │
  ├── Model performance
  │     ├── Confidence distribution (logprobs từ vLLM)
  │     ├── Emotion label distribution over time
  │     └── Response latency
  │
  ├── Data drift detection
  │     ├── Input text distribution thay đổi?
  │     ├── Từ vựng mới xuất hiện?
  │     └── User behavior pattern thay đổi?
  │
  ├── Business metrics
  │     ├── User engagement after emotion response
  │     ├── Teacher satisfaction (nếu có)
  │     └── Session completion rate
  │
  └── Alerts
        ├── Confidence trung bình drop > 5% → trigger retrain
        ├── 1 class chiếm > 60% predictions → investigate
        └── Error rate > threshold → rollback ngay
```

**Feedback loop:**
```
Low confidence predictions 
  → Log lại input + output
    → Feed vào auto-labeling pipeline (Giai đoạn 1)
      → Tích lũy đủ → trigger retrain (Giai đoạn 2)
        → Eval → Decide → Deploy → Monitor
          → Repeat ♻️
```

## Nhóm 1: End-to-End — Gần full pipeline nhất

### 🥇 NVIDIA NeMo + Data Flywheel Blueprint

Đây là giải pháp **gần nhất** với những gì bạn mô tả. NVIDIA NeMo microservices cung cấp nền tảng end-to-end để xây dựng data flywheel, cho phép doanh nghiệp liên tục tối ưu AI agents bằng cách curate data, customize LLM, và evaluate models. [NVIDIA Developer](https://developer.nvidia.com/blog/enhance-your-ai-agent-with-data-flywheels-using-nvidia-nemo-microservices/)

**Gồm các thành phần:**

- **NeMo Curator** — GPU-accelerated data curation
- **NeMo Customizer** — Fine-tune bằng LoRA, p-tuning, SFT qua 1 API call
- **NeMo Evaluator** — Auto eval với 20+ benchmarks chuẩn + custom benchmarks + LLM-as-judge
- **NeMo Guardrails** — Safety layer
- **NIM** — Serving inference (tương tự vLLM)

Blueprint này cung cấp giải pháp tự động, bán tự trị — vận hành không cần labeled data hay human-in-the-loop curation. Nó tự đánh giá models bằng custom metrics và task-specific benchmarks, dùng LLM làm automated judges. [NVIDIA](https://build.nvidia.com/nvidia/build-an-enterprise-data-flywheel/blueprintcard)

Kết quả thực tế: Flywheel phát hiện rằng task Tool Calling đơn giản đến mức một model fine-tuned llama-3.2-1b-instruct đạt ~98% accuracy so với model 70B đang chạy production. [GitHub](https://github.com/NVIDIA-AI-Blueprints/data-flywheel)

**Tuy nhiên**, NVIDIA cũng thừa nhận: Flywheel nên được xem như đèn pin soi đường, không phải autopilot — việc promote lên production vẫn là quyết định của con người. [GitHub](https://github.com/NVIDIA-AI-Blueprints/data-flywheel)

**Nhược điểm:** Yêu cầu NVIDIA GPU, Kubernetes, DGX Cloud — khá nặng cho team nhỏ. Nhưng concept và architecture thì rất đáng học theo.

### 🥈 MLRun (Iguazio) + NVIDIA NeMo

MLRun đã tích hợp với NVIDIA NeMo microservices để mở rộng Data Flywheel Blueprint. Integration này cho phép tự động train, evaluate, fine-tune và monitor AI models ở quy mô lớn. [Mlrun](https://www.mlrun.org/blog/mlrun-nvidia-nemo-building-observable-ai-data-flywheels-in-production/)

MLRun đóng vai trò flywheel orchestrator — bao bọc flywheel và điều phối training, fine-tuning theo use case cụ thể, evaluation và monitoring. Nếu cần quyết định từ human-in-the-loop, MLRun sẽ orchestrate feedback đó đến ứng dụng đã deploy. [Iguazio](https://www.iguazio.com/blog/observable-data-flywheels-for-production-with-mlrun-and-nvidia-nemo/)

### Data & Labeling

Có thể apply recipes chỉ với YAML file mà không cần viết code. [Microsoft Community Hub](https://techcommunity.microsoft.com/blog/azure-ai-foundry-blog/fine-tuneevaluatequantize-slmllm-using-the-torchtune-on-azure-ml/4285663)

**MLflow** — Model registry, experiment tracking, artifact management. Open-source, mature.

**Weights & Biases** — Experiment tracking, hyperparameter sweeps, model registry. Có free tier.

**Langfuse** — Datasets cho phép tạo test sets và benchmarks để evaluate LLM application, hỗ trợ continuous improvement, pre-deployment testing, và tích hợp với LangChain/LlamaIndex. [The OSS LLMOps Stack](https://oss-llmops-stack.com/) Bạn đang dùng Langfuse rồi nên đây là lợi thế.

| **Answer** | Xây Continuous Training (CT) Pipeline với 5 steps tự động, sử dụng Distilabel + Argilla (labeling), Unsloth (training), sklearn (eval), MLflow (registry). |

### Business Context

- **Product**: Pika Robot — robot giáo dục cho trẻ em
- **Emotion API**: Phân loại cảm xúc từ text → 13 nhóm emotion → robot thể hiện animation tương ứng
- **Current model**: Qwen2.5-1.5B-Instruct-AWQ (base, chưa fine-tune)

## 5. 3-Way Agreement Logic

```
┌───────────────────────────────────────────────────────────┐
│                    3-WAY AGREEMENT                         │
│                                                            │
│  Sources: AI_label, Human_label, Model_output              │
│                                                            │
│  Case 1: AI == Human == Model                              │
│    → auto_approved (high confidence)                       │
│    → Dùng luôn làm training label                         │
│                                                            │
│  Case 2: AI == Human != Model                              │
│    → auto_approved (model đang sai, cần improve)          │
│    → GOLD — đây là data giá trị nhất để train             │
│                                                            │
│  Case 3: AI == Model != Human                              │
│    → Trust human → human_label là ground truth             │
│    → Flag AI labeling prompt cần improve                   │
│                                                            │
│  Case 4: AI != Human (bất kể Model)                        │
│    → flagged → cần thêm human review                      │
│    → Không dùng cho training cho đến khi resolved          │
│                                                            │
│  Case 5: Chỉ có AI + Model (chưa có Human)                │
│    → AI == Model: auto_approved (low confidence)           │
│    → AI != Model: pending → cần human review              │
└───────────────────────────────────────────────────────────┘
```

## 6. Training Strategy

### Base Principle

```
LUÔN train từ BASE MODEL gốc:
  Qwen/Qwen2.5-1.5B-Instruct  (HuggingFace)
  
KHÔNG BAO GIỜ train từ model đã fine-tune trước đó.

Lý do:
  - Tránh catastrophic forgetting tích lũy
  - 100% reproducible
  - Dễ debug (chỉ 1 biến: data)
```

### Training Config

```yaml
base_model: "Qwen/Qwen2.5-1.5B-Instruct"
method: "lora"
lora:
  r: 16
  alpha: 32
  target_modules: ["q_proj", "k_proj", "v_proj", "o_proj"]
  dropout: 0.05
training:
  epochs: 3
  batch_size: 8
  learning_rate: 2e-4
  warmup_ratio: 0.1
  max_seq_length: 512
quantization:
  output_format: "awq"
  bits: 4
```

## 7. Evaluation Strategy

### Metrics

| Metric | Mục đích | Threshold |
|--------|----------|-----------|
| Accuracy | Overall correctness | ≥ baseline + 0.5% |
| F1 Macro | Balance across classes | ≥ baseline + 0.3% |
| Per-class F1 | Không class nào bị sacrifice | Drop < 2% |
| Confusion Matrix | Xem model nhầm lẫn ở đâu | Visual review |
| Regression Pass Rate | Edge cases đã fix | 100% |
| Inference Latency | Không chậm hơn | P95 < 100ms |

### 3 Test Sets

| Set | Size | Source | Thay đổi? |
|-----|------|--------|-----------|
| Benchmark | ≥500 | Random from labeled data | Cố định per version |
| Regression | Tăng dần | Edge cases đã phát hiện | Chỉ thêm, không xóa |

from ..entities import EmotionSample
from ..value_objects import AgreementStatus, EmotionLabel


class LabelAgreementService:
    """3-way agreement logic: AI vs Human vs Model output."""

    def resolve(self, sample: EmotionSample) -> EmotionSample:
        ai = sample.ai_label
        human = sample.human_label
        model = sample.model_output

        # Case 5: chỉ có AI + Model (chưa có human)
        if human is None:
            if ai == model:
                sample.agreed_label = ai
                sample.agreement_status = AgreementStatus.AUTO_APPROVED
            else:
                sample.agreement_status = AgreementStatus.PENDING
            return sample

        # Case 1: all 3 agree
        if ai == human == model:
            sample.agreed_label = human
            sample.agreement_status = AgreementStatus.AUTO_APPROVED
            return sample

        # Case 2: AI == Human != Model (model sai, valuable training data)
        if ai == human and ai != model:
            sample.agreed_label = human
            sample.agreement_status = AgreementStatus.AUTO_APPROVED
            return sample

        # Case 3: AI == Model != Human (trust human)
        if ai == model and ai != human:
            sample.agreed_label = human
            sample.agreement_status = AgreementStatus.HUMAN_RESOLVED
            return sample

        # Case 4: AI != Human (conflict)
        sample.agreement_status = AgreementStatus.FLAGGED
        return sample

    def batch_resolve(
        self, samples: list[EmotionSample]
    ) -> tuple[list[EmotionSample], list[EmotionSample]]:
        """Returns (approved, flagged)."""
        approved, flagged = [], []
        for s in samples:
            resolved = self.resolve(s)
            if resolved.agreement_status in (
                AgreementStatus.AUTO_APPROVED,
                AgreementStatus.HUMAN_RESOLVED,
            ):
                approved.append(resolved)
            else:
                flagged.append(resolved)
        return approved, flagged
```

### 3.3 promotion_decider.py

```python
from dataclasses import dataclass
from ..entities import EvalResult


@dataclass
class PromotionPolicy:
    min_accuracy_improvement: float = 0.005   # +0.5%
    min_f1_improvement: float = 0.003         # +0.3%
    max_per_class_regression: float = 0.02    # -2%
    min_regression_pass_rate: float = 1.0     # 100%
    min_test_set_size: int = 500


class PromotionDeciderService:

    def __init__(self, policy: PromotionPolicy):
        self.policy = policy

    def decide(
        self,
        candidate: EvalResult,
        baseline: EvalResult,
    ) -> tuple[bool, str]:
        """Returns (should_promote, reason)."""
        p = self.policy

        # Check accuracy
        acc_diff = candidate.accuracy - baseline.accuracy
        if acc_diff < p.min_accuracy_improvement:
            return False, (
                f"Accuracy +{acc_diff:.4f} < threshold +{p.min_accuracy_improvement}"
            )

        # Check F1 macro
        f1_diff = candidate.f1_macro - baseline.f1_macro
        if f1_diff < p.min_f1_improvement:
            return False, (
                f"F1 macro +{f1_diff:.4f} < threshold +{p.min_f1_improvement}"
            )

        # Check per-class regression
        for cls, f1 in candidate.f1_per_class.items():
            baseline_f1 = baseline.f1_per_class.get(cls, 0)
            drop = baseline_f1 - f1
            if drop > p.max_per_class_regression:
                return False, (
                    f"Class '{cls}' F1 dropped {drop:.4f} > max {p.max_per_class_regression}"
                )

        # Check regression tests
        if candidate.regression_pass_rate < p.min_regression_pass_rate:
            return False, (
                f"Regression pass {candidate.regression_pass_rate:.0%} "
                f"< required {p.min_regression_pass_rate:.0%}"
            )

        # Check test set size
        if candidate.benchmark_size < p.min_test_set_size:
            return False, (
                f"Test set {candidate.benchmark_size} < min {p.min_test_set_size}"
            )

        return True, "All criteria passed"

                model_path, regression_path

## Best practices trong việc thiết kế 1 hệ thống tự finetune tự deploy, tự đánh giá kết quả, tự chốt xem có lên phiên bản hay ko, hoàn toàn tự động, ...

Đây là câu hỏi rất hay — bạn đang muốn xây **Continuous Training Pipeline** (CT Pipeline) cho SLM. Đây là bài toán MLOps/LLMOps cấp production. Mình sẽ thiết kế từ tổng quan đến chi tiết.

---

### Bức tranh tổng thể: Closed-Loop LLMOps

```
┌─────────────────────────────────────────────────────────────────┐
│                    CLOSED-LOOP CT PIPELINE                       │
│                                                                  │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 1. DATA  │───▶│ 2. TRAIN │───▶│ 3. EVAL  │───▶│4. DECIDE │  │
│  │ COLLECT  │    │          │    │          │    │ PROMOTE? │  │
│  └──────────┘    └──────────┘    └──────────┘    └────┬─────┘  │
│       ▲                                               │         │
│       │           ┌──────────┐    ┌──────────┐        │         │
│       │           │ 6. MON   │◀───│ 5. DEPLOY│◀───────┘         │
│       │           │ ITOR     │    │ (canary) │   YES             │
│       └───────────┴──────────┘    └──────────┘                  │
│            feedback loop                                         │
└─────────────────────────────────────────────────────────────────┘
```

Gồm 6 giai đoạn, mỗi giai đoạn chạy tự động và trigger giai đoạn tiếp theo.

---

### Giai đoạn 1: Data Collection & Auto-Labeling

**Mục tiêu:** Liên tục thu thập data mới từ production, tự gán nhãn, tự kiểm tra chất lượng.

```
Production traffic (emotion-api)
  │
  ▼
Log collector (ghi lại input + model output + user feedback)
  │
  ▼
Auto-labeler (LLM mạnh gán nhãn, multi-vote)
  │
  ▼
Quality gate (lọc confidence thấp → human review queue)
  │
  ▼
Data store (versioned dataset - DVC hoặc HuggingFace Dataset)
```

**Các nguồn signal quan trọng:**

- **Implicit feedback:** User có tiếp tục tương tác sau khi bot respond theo emotion không? Nếu bot detect "happy" mà user bỏ cuộc → có thể sai
- **Explicit feedback:** Nếu có nút 👍👎 hoặc teacher đánh giá
- **Hard examples:** Những sample mà model hiện tại confidence thấp (logprobs từ vLLM) → đây là gold mine để improve
- **Distribution shift:** Input mới khác pattern cũ (new slang, new topics)

**Auto-labeling pipeline:**

```
Mỗi ngày/tuần chạy batch:
1. Lấy N samples mới từ production logs
2. Gọi Claude/GPT gán nhãn (có reasoning + confidence)
3. Multi-vote nếu budget cho phép
4. Samples confidence HIGH → auto-add vào training pool
5. Samples confidence LOW → đẩy vào human review queue
6. Tính label distribution → alert nếu drift
```

**Trigger chuyển sang giai đoạn 2:**

- Khi accumulated data mới đạt threshold (vd: 500+ samples mới)
- Hoặc theo schedule (weekly/bi-weekly)
- Hoặc khi monitoring phát hiện model degradation

### Giai đoạn 2: Auto Training

**Mục tiêu:** Tự động fine-tune khi có đủ data mới, reproducible 100%.

```
Trigger (đủ data / schedule / quality drop)
  │
  ▼
Data preparation
  ├── Merge new data + existing validated data
  ├── Train/Val/Test split (stratified by label)
  ├── Data versioning (tag: v2.3.1)
  │
  ▼
Training job
  ├── Base: luôn từ Qwen2.5-1.5B-Instruct (KHÔNG từ model cũ đã fine-tune)
  ├── Method: LoRA/QLoRA
  ├── Hyperparams: từ config file (versioned)
  ├── Hardware: auto-provision GPU (RunPod / on-prem)
  │
  ▼
Artifacts
  ├── LoRA adapter weights
  ├── Training metrics (loss curve, lr schedule)
  ├── Training config snapshot
  └── Data version reference
```

**Best practices training:**

- **Luôn train từ base model gốc**, không train incremental từ checkpoint cũ. Lý do: tránh catastrophic forgetting tích lũy, dễ reproduce, dễ debug
- **Toàn bộ config là code** — version control trên Git, không manual tweak
- **Mỗi training run có unique ID**, log đầy đủ lên experiment tracker (MLflow, Weights & Biases, hoặc đơn giản là structured logs)

**Anti-pattern cần tránh:**

```
❌ Train tiếp từ model fine-tuned lần trước
   → Catastrophic forgetting tích lũy, model ngày càng tệ

❌ Merge data mới mà không validate 
   → Garbage in, garbage out

❌ Dùng hyperparams cố định mãi
   → Data thay đổi, optimal hyperparams cũng thay đổi
   → Có thể chạy nhẹ hyperparameter search mỗi N lần train
```

### Giai đoạn 3: Auto Evaluation (Gate quan trọng nhất)

**Mục tiêu:** Tự động đánh giá model mới, so sánh với model đang chạy production, quyết định có promote hay không.

```
Model mới (candidate)
  │
  ▼
Eval suite chạy tự động
  │
  ├── Benchmark test set (cố định, không bao giờ dùng để train)
  │     ├── Accuracy / F1 / Precision / Recall per class
  │     ├── Confusion matrix
  │     └── Latency (inference speed)
  │
  ├── Regression test set (edge cases đã từng fix)
  │     └── "Những case model cũ từng sai, đã fix"
  │         → Model mới PHẢI pass hết
  │
  ├── Golden test set (human-curated, high quality)
  │     └── 100-200 samples, gán nhãn bởi expert
  │         → Đây là "source of truth"
  │
  ├── A/B comparison với model hiện tại (production)
  │     └── Cùng input → so output
  │
  └── Safety checks
        ├── Không output toxic content
        ├── Không hallucinate emotion
        └── Latency < threshold (cho vLLM serving)
```

**Promotion criteria (ví dụ):**

```yaml
promotion_rules:
  ## Model mới phải tốt hơn model cũ
  accuracy_improvement_min: 0.5%    ## ít nhất +0.5%
  f1_improvement_min: 0.3%
  
  ## Không được tệ hơn ở bất kỳ class nào
  per_class_regression_max: -2%     ## không class nào drop > 2%
  
  ## Regression tests
  regression_test_pass_rate: 100%   ## phải pass hết
  
  ## Performance
  p95_latency_max_ms: 100          ## không chậm hơn
  
  ## Safety
  safety_check: pass
  
  ## Decision
  auto_promote: true                ## nếu tất cả pass → tự deploy
  require_human_approval: false     ## true nếu muốn thêm 1 bước manual
```

**3 loại test set phải tách biệt — rất quan trọng:**

```
┌─────────────────────────────────────────────────┐
│ Training data     → dùng để train               │
│ Validation data   → dùng để tune hyperparams    │
│ Benchmark test    → CỐ ĐỊNH, đo performance     │
│ Regression test   → TÍCH LŨY edge cases đã fix  │
│ Golden test       → HUMAN-CURATED, source truth  │
└─────────────────────────────────────────────────┘

KHÔNG BAO GIỜ để test data lọt vào training data.
```

**Regression test set đặc biệt quan trọng:** Mỗi khi phát hiện model sai 1 case → fix xong → thêm case đó vào regression test. Set này chỉ tăng, không giảm. Nó đảm bảo model mới không "quên" những gì đã fix.

### Giai đoạn 4: Auto Decision — Promote hay Reject

```
Eval results
  │
  ▼
Decision engine
  │
  ├── ALL criteria pass?
  │     ├── YES → Tag model (v2.3.1), promote to staging
  │     └── NO  → Log reason, alert team, giữ model cũ
  │
  ├── Edge case: model mới tốt hơn overall nhưng 
  │   1 class bị regression?
  │     └── Alert team, KHÔNG auto-promote
  │         (cần human review)
  │
  └── Edge case: model mới ngang bằng model cũ?
        └── KHÔNG promote (không có lý do thay đổi)
```

**Principle: conservative by default** — chỉ promote khi chắc chắn tốt hơn. Khi nghi ngờ → giữ model cũ → alert team.

### Giai đoạn 5: Auto Deploy (Canary/Blue-Green)

**Đây là nơi connect với docker-compose hiện tại của bạn:**

```
Model promoted
  │
  ▼
Quantize (AWQ) tự động
  │
  ▼
Push model artifact lên registry (HF / S3 / local NFS)
  │
  ▼
Canary deployment
  ├── 10% traffic → model mới (vllm-qwen-canary)
  ├── 90% traffic → model cũ  (vllm-qwen-stable)
  │
  ▼
Monitor 2-4 giờ
  ├── Error rate tăng?  → auto rollback
  ├── Latency tăng?     → auto rollback
  ├── Emotion distribution lệch bất thường? → auto rollback
  ├── Tất cả OK?        → ramp lên 50% → 100%
  │
  ▼

  └── Model cũ giữ lại làm rollback backup
```

### Giai đoạn 6: Monitoring & Feedback Loop

**Đây là thứ biến pipeline từ "chạy 1 lần" thành "tự cải thiện liên tục":**

```
Production model đang serve
  │
  ▼
Monitor liên tục
  │
  ├── Model performance
  │     ├── Confidence distribution (logprobs từ vLLM)
  │     ├── Emotion label distribution over time
  │     └── Response latency
  │
  ├── Data drift detection
  │     ├── Input text distribution thay đổi?
  │     ├── Từ vựng mới xuất hiện?
  │     └── User behavior pattern thay đổi?
  │
  ├── Business metrics
  │     ├── User engagement after emotion response
  │     ├── Teacher satisfaction (nếu có)
  │     └── Session completion rate
  │
  └── Alerts
        ├── Confidence trung bình drop > 5% → trigger retrain
        ├── 1 class chiếm > 60% predictions → investigate
        └── Error rate > threshold → rollback ngay
```

**Feedback loop:**

```
Low confidence predictions 
  → Log lại input + output
    → Feed vào auto-labeling pipeline (Giai đoạn 1)
      → Tích lũy đủ → trigger retrain (Giai đoạn 2)
        → Eval → Decide → Deploy → Monitor
          → Repeat ♻️
```

Blueprint này cung cấp giải pháp tự động, bán tự trị — vận hành không cần labeled data hay human-in-the-loop curation. Nó tự đánh giá models bằng custom metrics và task-specific benchmarks, dùng LLM làm automated judges.

Kết quả thực tế: Flywheel phát hiện rằng task Tool Calling đơn giản đến mức một model fine-tuned llama-3.2-1b-instruct đạt ~98% accuracy so với model 70B đang chạy production.

**Tuy nhiên**, NVIDIA cũng thừa nhận: Flywheel nên được xem như đèn pin soi đường, không phải autopilot — việc promote lên production vẫn là quyết định của con người.

**Nhược điểm:** Yêu cầu NVIDIA GPU, Kubernetes, DGX Cloud — khá nặng cho team nhỏ. Nhưng concept và architecture thì rất đáng học theo.

MLRun đã tích hợp với NVIDIA NeMo microservices để mở rộng Data Flywheel Blueprint. Integration này cho phép tự động train, evaluate, fine-tune và monitor AI models ở quy mô lớn.

MLRun đóng vai trò flywheel orchestrator — bao bọc flywheel và điều phối training, fine-tuning theo use case cụ thể, evaluation và monitoring. Nếu cần quyết định từ human-in-the-loop, MLRun sẽ orchestrate feedback đó đến ứng dụng đã deploy.

Đây là combo mạnh nhất hiện tại cho production.

### Nhóm 2: Orchestration — Xương sống pipeline

Các tools này không train model trực tiếp nhưng **orchestrate toàn bộ flow**: trigger, chạy steps, track artifacts, quyết định promote.

#### ZenML

ZenML + OpenPipe integration cho phép khi chạy pipeline lần nữa với data mới, model sẽ tự động được retrain và redeploy, đảm bảo production model luôn phản ánh data mới nhất.

ZenML cho phép viết workflows (pipelines) chạy trên bất kỳ infrastructure backend nào. Nó tự động containerize và track code, track từng run với metrics/logs/metadata, và tích hợp với tools hiện có như MLflow, LangGraph, Langfuse, SageMaker, GCP Vertex.

**Ưu điểm lớn:** Open-source (Apache 2.0), infrastructure-agnostic, có sẵn examples cho LLM fine-tuning pipeline. **Phù hợp nhất với team bạn** vì nhẹ, flexible, chạy được local.

#### Dagster

Dagster cung cấp automation pipeline bao gồm refresh data và models theo nhu cầu business, khả năng thử nghiệm hyperparameters khác nhau qua resources, và quản lý environment cho phép test ở dev rồi deploy lên production seamlessly.

Dagster thiên về data engineering hơn nhưng vẫn dùng tốt cho ML pipelines.

#### Prefect / Airflow

Traditional orchestrators — vẫn dùng được nhưng không có tính năng ML-native (artifact tracking, model registry) như ZenML hay Dagster.

### Nhóm 3: Specialized — Từng giai đoạn cụ thể

#### Data & Labeling

**Argilla** — Argilla là collaboration tool cho AI engineers và domain experts để xây high-quality datasets. Approach lập trình của Argilla cho phép xây workflows cho continuous evaluation và model improvement.

**Distilabel (by Argilla)** — Framework cho synthetic data và AI feedback, tích hợp AI feedback từ bất kỳ LLM provider nào qua unified API. Synthesize và judge data theo latest research papers.

#### Evaluation

**DeepEval** — Framework evaluation cho LLM hỗ trợ cả end-to-end và component-level evaluation, với nhiều metrics sẵn dùng powered by LLM, statistical methods, hoặc NLP models chạy local.

**Opik (by Comet)** — Open-source platform cho evaluating, testing, và monitoring LLM applications. Hỗ trợ log traces trong development và production, chạy experiments với different prompts và evaluate against test sets.

#### Training

**Unsloth** — Tốt nhất cho Qwen2.5-1.5B (bạn đang dùng), 2x nhanh, 70% ít VRAM.

**Ludwig** — Low-code framework, mọi thứ cấu hình qua YAML declarative — architecture, training loop, hyperparameter search, và backend infrastructure đều có thể chỉnh qua config file. Hỗ trợ LoRA, QLoRA, auto batch size, distributed training.

**torchtune (PyTorch native)** — Library đơn giản hóa quá trình fine-tune và optimization, hỗ trợ distributed training, flexible logging, và model quantization. Có thể apply recipes chỉ với YAML file mà không cần viết code.

#### Experiment Tracking

**MLflow** — Model registry, experiment tracking, artifact management. Open-source, mature.

**Weights & Biases** — Experiment tracking, hyperparameter sweeps, model registry. Có free tier.

**Langfuse** — Datasets cho phép tạo test sets và benchmarks để evaluate LLM application, hỗ trợ continuous improvement, pre-deployment testing, và tích hợp với LangChain/LlamaIndex. Bạn đang dùng Langfuse rồi nên đây là lợi thế.

#### Step 2: AI Label + Human Label + Agreement

|Task|Tool|Chi tiết|
|---|---|---|
|AI gán nhãn|**Distilabel**|Gọi GPT-4o-mini/Claude Haiku gán nhãn hàng loạt|
|Human review|**Argilla**|UI để labeler xem và gán nhãn|
|Agreement check|**Custom logic**|`if AI == Human == User_Feedback → auto_approve`|

**Logic agreement bạn mô tả:**

```
IF AI_label == Human_label == User_feedback:
    → auto_approve (high confidence)
ELIF AI_label == Human_label != User_feedback:
    → flag để investigate (có thể user feedback sai)
ELIF AI_label != Human_label:
    → human review lại
```


###### Distilabel (AI Labeling)  + Langfuse để trace lại số tiền

Framework synthetic data/AI feedback, pipeline DAG đơn giản (context manager), tích hợp HF Datasets/LLM (Claude/GPT/OpenAI) unified API. https://github.com/argilla-io/distilabel

- **Dễ dùng**: Quickstart 10 dòng code → load dataset từ Langfuse JSON → TextGeneration step gọi GPT-4o-mini label emotion → push HF Hub.   http://distilabel.argilla.io/1.5.1/sections/getting_started/quickstart/
    Ví dụ emotion labeling:
    

```python
from distilabel.pipeline import Pipeline
from distilabel.steps import LoadDataFromHub, TextGeneration
from distilabel.llms import OpenAI

with Pipeline(name="emotion_labeler") as pipeline:
    load_data = LoadDataFromHub(name="load", dataset="your/langfuse-emotion-logs")
    label = TextGeneration(
        name="label",
        llm=OpenAI(model="gpt-4o-mini"),
        prompt="Classify emotion: {input_text}"
    )
## Run: distiset = pipeline.run()

```

Chạy local/Colab, fault-tolerant, scalable. Thời gian học: 30p docs/tutorial.[](https://argilla-io.github.io/distilabel/1.5.3/sections/getting_started/quickstart/)​
https://argilla-io.github.io/distilabel/1.5.3/sections/getting_started/quickstart/

###### Giải Thích Dễ Hiểu Như "Lego ML"

Hình dung pipeline emotion như dây chuyền nhà máy:

1. **Không ZenML**: Bạn viết 5 script riêng (extract.py, label.py, train.py...), cron chạy theo thứ tự → lỗi 1 step hỏng hết, khó track "run nào tốt nhất?".
    
2. **Có ZenML**: Xây "lego blocks" (@step) → ghép thành pipeline → `zenml run` tự chạy tuần tự/song song, track mọi thứ (data version, F1 score, LoRA file), UI compare "run 1 F1=0.92 vs run 2 F1=0.88".
    

**Vị trí cụ thể trong luồng bạn**:

```text
Raw logs (Langfuse)
  ↓ Step 1: zenml step "extract_data()"  ## Pull API → JSON dataset
Labeled data (Distilabel + Argilla)
  ↓ Step 2: zenml step "ai_label()" + "human_review()"
Train data → LoRA model (Unsloth)
  ↓ Step 3: zenml step "finetune_lora()"
Eval metrics (DeepEval)
  ↓ Step 4: zenml step "evaluate_f1()"
Model decision (MLflow)
  ↓ Step 5: zenml step "promote_deploy()" → vLLM canary

```

**Lợi ích đơn giản**:

- **Tự động**: Data mới → trigger rerun toàn pipeline (cron + webhook).
    
- **Track**: UI xem "LoRA run nào tốt nhất?", artifact download.
    
- **Portable**: Chạy local GPU hôm nay → AWS RunPod mai.
    

Ví dụ code siêu ngắn (5p test):

```python
from zenml import step, pipeline

@step  ## Lego block 1
def extract() -> str:
    return "emotion_logs.json"  ## Từ Langfuse

@step  ## Lego 2
def train(logs: str) -> str:
    return "lora_model"  ## Unsloth

@pipeline  ## Ghép lego
def emotion_flow():
    logs = extract()
    model = train(logs)

emotion_flow()  ## Chạy!

```

Với cách bạn đang làm — **lấy data bằng tay từ DataDog, export CSV, chạy script Python evaluate** — thì ZenML là **overkill hoàn toàn** cho giai đoạn này.

1. Quantize model mới thành AWQ
2. Push lên HuggingFace Hub (ví dụ: `your-org/Qwen2.5-1.5B-Emotion-v2-AWQ`)
3. Serving repo chỉ cần đổi 1 dòng trong docker-compose:

```yaml
## docker-compose.yml (serving repo)
command: >
  --model your-org/Qwen2.5-1.5B-Emotion-v2-AWQ    ## ← đổi model name
```

Hoặc dùng biến environment:

```yaml
command: >
  --model ${EMOTION_MODEL:-Qwen/Qwen2.5-1.5B-Instruct-AWQ}
```

### Thứ duy nhất cần share: Emotion Labels

13 emotion groups cần giống nhau giữa 2 repo. 3 cách:

| Cách | Pros | Cons |
|---|---|---|
| **Python package nhỏ** (`pip install emotion-shared`) | Clean nhất, version control | Overhead maintain package |
| **Config file** (YAML/JSON) trong cả 2 repo | Đơn giản | Phải sync manual |
| **Git submodule** | Auto sync | Git submodule phức tạp |

**Cho Phase 1: copy YAML file là đủ.** 13 emotion groups ít khi thay đổi. Khi nào thay đổi thường xuyên thì upgrade lên shared package.
