# Best practices trong việc thiết kế 1 hệ thống tự finetune tự deploy, tự đánh giá kết quả, tự chốt xem có lên phiên bản hay ko, hoàn toàn tự động, ...

Đây là câu hỏi rất hay — bạn đang muốn xây **Continuous Training Pipeline** (CT Pipeline) cho SLM. Đây là bài toán MLOps/LLMOps cấp production. Mình sẽ thiết kế từ tổng quan đến chi tiết.

---

## Bức tranh tổng thể: Closed-Loop LLMOps

```
┌─────────────────────────────────────────────────────────────────┐
│                    CLOSED-LOOP CT PIPELINE                       │
│                                                                  │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 1. DATA  │───▶│ 2. TRAIN │───▶│ 3. EVAL  │───▶│4. DECIDE │  │
│  │ COLLECT  │    │          │    │          │    │ PROMOTE? │  │
│  └──────────┘    └──────────┘    └──────────┘    └────┬─────┘  │
│       ▲                                               │         │
│       │           ┌──────────┐    ┌──────────┐        │         │
│       │           │ 6. MON   │◀───│ 5. DEPLOY│◀───────┘         │
│       │           │ ITOR     │    │ (canary) │   YES             │
│       └───────────┴──────────┘    └──────────┘                  │
│            feedback loop                                         │
└─────────────────────────────────────────────────────────────────┘
```

Gồm 6 giai đoạn, mỗi giai đoạn chạy tự động và trigger giai đoạn tiếp theo.

## Giai đoạn 1: Data Collection & Auto-Labeling

**Mục tiêu:** Liên tục thu thập data mới từ production, tự gán nhãn, tự kiểm tra chất lượng.

```
Production traffic (emotion-api)
  │
  ▼
Log collector (ghi lại input + model output + user feedback)
  │
  ▼
Auto-labeler (LLM mạnh gán nhãn, multi-vote)
  │
  ▼
Quality gate (lọc confidence thấp → human review queue)
  │
  ▼
Data store (versioned dataset - DVC hoặc HuggingFace Dataset)
```

**Các nguồn signal quan trọng:**

- **Implicit feedback:** User có tiếp tục tương tác sau khi bot respond theo emotion không? Nếu bot detect "happy" mà user bỏ cuộc → có thể sai
- **Explicit feedback:** Nếu có nút 👍👎 hoặc teacher đánh giá
- **Hard examples:** Những sample mà model hiện tại confidence thấp (logprobs từ vLLM) → đây là gold mine để improve
- **Distribution shift:** Input mới khác pattern cũ (new slang, new topics)

**Auto-labeling pipeline:**

```
Mỗi ngày/tuần chạy batch:
1. Lấy N samples mới từ production logs
2. Gọi Claude/GPT gán nhãn (có reasoning + confidence)
3. Multi-vote nếu budget cho phép
4. Samples confidence HIGH → auto-add vào training pool
5. Samples confidence LOW → đẩy vào human review queue
6. Tính label distribution → alert nếu drift
```

**Trigger chuyển sang giai đoạn 2:**

- Khi accumulated data mới đạt threshold (vd: 500+ samples mới)
- Hoặc theo schedule (weekly/bi-weekly)
- Hoặc khi monitoring phát hiện model degradation

## Giai đoạn 2: Auto Training

**Mục tiêu:** Tự động fine-tune khi có đủ data mới, reproducible 100%.

```
Trigger (đủ data / schedule / quality drop)
  │
  ▼
Data preparation
  ├── Merge new data + existing validated data
  ├── Train/Val/Test split (stratified by label)
  ├── Data versioning (tag: v2.3.1)
  │
  ▼
Training job
  ├── Base: luôn từ Qwen2.5-1.5B-Instruct (KHÔNG từ model cũ đã fine-tune)
  ├── Method: LoRA/QLoRA
  ├── Hyperparams: từ config file (versioned)
  ├── Hardware: auto-provision GPU (RunPod / on-prem)
  │
  ▼
Artifacts
  ├── LoRA adapter weights
  ├── Training metrics (loss curve, lr schedule)
  ├── Training config snapshot
  └── Data version reference
```

**Best practices training:**

- **Luôn train từ base model gốc**, không train incremental từ checkpoint cũ. Lý do: tránh catastrophic forgetting tích lũy, dễ reproduce, dễ debug
- **Toàn bộ config là code** — version control trên Git, không manual tweak
- **Mỗi training run có unique ID**, log đầy đủ lên experiment tracker (MLflow, Weights & Biases, hoặc đơn giản là structured logs)

**Anti-pattern cần tránh:**

```
❌ Train tiếp từ model fine-tuned lần trước
   → Catastrophic forgetting tích lũy, model ngày càng tệ

❌ Merge data mới mà không validate 
   → Garbage in, garbage out

❌ Dùng hyperparams cố định mãi
   → Data thay đổi, optimal hyperparams cũng thay đổi
   → Có thể chạy nhẹ hyperparameter search mỗi N lần train
```

## Giai đoạn 3: Auto Evaluation (Gate quan trọng nhất)

**Mục tiêu:** Tự động đánh giá model mới, so sánh với model đang chạy production, quyết định có promote hay không.

```
Model mới (candidate)
  │
  ▼
Eval suite chạy tự động
  │
  ├── Benchmark test set (cố định, không bao giờ dùng để train)
  │     ├── Accuracy / F1 / Precision / Recall per class
  │     ├── Confusion matrix
  │     └── Latency (inference speed)
  │
  ├── Regression test set (edge cases đã từng fix)
  │     └── "Những case model cũ từng sai, đã fix"
  │         → Model mới PHẢI pass hết
  │
  ├── Golden test set (human-curated, high quality)
  │     └── 100-200 samples, gán nhãn bởi expert
  │         → Đây là "source of truth"
  │
  ├── A/B comparison với model hiện tại (production)
  │     └── Cùng input → so output
  │
  └── Safety checks
        ├── Không output toxic content
        ├── Không hallucinate emotion
        └── Latency < threshold (cho vLLM serving)
```

**Promotion criteria (ví dụ):**

```yaml
promotion_rules:
  # Model mới phải tốt hơn model cũ
  accuracy_improvement_min: 0.5%    # ít nhất +0.5%
  f1_improvement_min: 0.3%
  
  # Không được tệ hơn ở bất kỳ class nào
  per_class_regression_max: -2%     # không class nào drop > 2%
  
  # Regression tests
  regression_test_pass_rate: 100%   # phải pass hết
  
  # Performance
  p95_latency_max_ms: 100          # không chậm hơn
  
  # Safety
  safety_check: pass
  
  # Decision
  auto_promote: true                # nếu tất cả pass → tự deploy
  require_human_approval: false     # true nếu muốn thêm 1 bước manual
```

**3 loại test set phải tách biệt — rất quan trọng:**

```
┌─────────────────────────────────────────────────┐
│ Training data     → dùng để train               │
│ Validation data   → dùng để tune hyperparams    │
│ Benchmark test    → CỐ ĐỊNH, đo performance     │
│ Regression test   → TÍCH LŨY edge cases đã fix  │
│ Golden test       → HUMAN-CURATED, source truth  │
└─────────────────────────────────────────────────┘

KHÔNG BAO GIỜ để test data lọt vào training data.
```

**Regression test set đặc biệt quan trọng:** Mỗi khi phát hiện model sai 1 case → fix xong → thêm case đó vào regression test. Set này chỉ tăng, không giảm. Nó đảm bảo model mới không "quên" những gì đã fix.

## Giai đoạn 4: Auto Decision — Promote hay Reject

```
Eval results
  │
  ▼
Decision engine
  │
  ├── ALL criteria pass?
  │     ├── YES → Tag model (v2.3.1), promote to staging
  │     └── NO  → Log reason, alert team, giữ model cũ
  │
  ├── Edge case: model mới tốt hơn overall nhưng 
  │   1 class bị regression?
  │     └── Alert team, KHÔNG auto-promote
  │         (cần human review)
  │
  └── Edge case: model mới ngang bằng model cũ?
        └── KHÔNG promote (không có lý do thay đổi)
```

**Principle: conservative by default** — chỉ promote khi chắc chắn tốt hơn. Khi nghi ngờ → giữ model cũ → alert team.

## Giai đoạn 5: Auto Deploy (Canary/Blue-Green)

**Đây là nơi connect với docker-compose hiện tại của bạn:**

```
Model promoted
  │
  ▼
Quantize (AWQ) tự động
  │
  ▼
Push model artifact lên registry (HF / S3 / local NFS)
  │
  ▼
Canary deployment
  ├── 10% traffic → model mới (vllm-qwen-canary)
  ├── 90% traffic → model cũ  (vllm-qwen-stable)
  │
  ▼
Monitor 2-4 giờ
  ├── Error rate tăng?  → auto rollback
  ├── Latency tăng?     → auto rollback
  ├── Emotion distribution lệch bất thường? → auto rollback
  ├── Tất cả OK?        → ramp lên 50% → 100%
  │
  ▼
Full rollout → model mới thành stable
  └── Model cũ giữ lại làm rollback backup
```

**Với docker-compose, cách đơn giản nhất:**

```
# Blue-Green: 2 vLLM containers
vllm-qwen-stable:   # model hiện tại (luôn chạy)
vllm-qwen-canary:   # model mới (deploy khi có candidate)

# Load balancer (nginx/traefik) 
# điều traffic theo tỷ lệ canary
```

## Giai đoạn 6: Monitoring & Feedback Loop

**Đây là thứ biến pipeline từ "chạy 1 lần" thành "tự cải thiện liên tục":**

```
Production model đang serve
  │
  ▼
Monitor liên tục
  │
  ├── Model performance
  │     ├── Confidence distribution (logprobs từ vLLM)
  │     ├── Emotion label distribution over time
  │     └── Response latency
  │
  ├── Data drift detection
  │     ├── Input text distribution thay đổi?
  │     ├── Từ vựng mới xuất hiện?
  │     └── User behavior pattern thay đổi?
  │
  ├── Business metrics
  │     ├── User engagement after emotion response
  │     ├── Teacher satisfaction (nếu có)
  │     └── Session completion rate
  │
  └── Alerts
        ├── Confidence trung bình drop > 5% → trigger retrain
        ├── 1 class chiếm > 60% predictions → investigate
        └── Error rate > threshold → rollback ngay
```

**Feedback loop:**

```
Low confidence predictions 
  → Log lại input + output
    → Feed vào auto-labeling pipeline (Giai đoạn 1)
      → Tích lũy đủ → trigger retrain (Giai đoạn 2)
        → Eval → Decide → Deploy → Monitor
          → Repeat ♻️
```

## Bắt đầu từ đâu? (Phased approach)

Đừng xây hết 1 lần. Chia thành phases:

**Phase 1 (tuần 1-2):** Manual pipeline nhưng scripted. Train bằng script, eval bằng script, deploy bằng tay. Mục tiêu: có 1 model fine-tuned chạy production.

**Phase 2 (tuần 3-4):** Semi-auto. Auto-label + auto-train + auto-eval. Nhưng promote/deploy vẫn cần human approve.

**Phase 3 (tháng 2):** Full auto. Thêm canary deploy, auto-promote, monitoring + feedback loop. Lúc này pipeline tự xoay.

---

Bạn muốn mình đi sâu vào giai đoạn nào trước? Mình nghĩ nên bắt đầu từ **Phase 1** — thiết kế eval suite trước (vì nếu không đo được thì không improve được), rồi mới tính chuyện auto-train và auto-deploy.

## Nhóm 1: End-to-End — Gần full pipeline nhất

### 🥇 NVIDIA NeMo + Data Flywheel Blueprint

Đây là giải pháp **gần nhất** với những gì bạn mô tả. NVIDIA NeMo microservices cung cấp nền tảng end-to-end để xây dựng data flywheel, cho phép doanh nghiệp liên tục tối ưu AI agents bằng cách curate data, customize LLM, và evaluate models.

**Gồm các thành phần:**

- **NeMo Curator** — GPU-accelerated data curation
- **NeMo Customizer** — Fine-tune bằng LoRA, p-tuning, SFT qua 1 API call
- **NeMo Evaluator** — Auto eval với 20+ benchmarks chuẩn + custom benchmarks + LLM-as-judge
- **NeMo Guardrails** — Safety layer
- **NIM** — Serving inference (tương tự vLLM)

Blueprint này cung cấp giải pháp tự động, bán tự trị — vận hành không cần labeled data hay human-in-the-loop curation. Nó tự đánh giá models bằng custom metrics và task-specific benchmarks, dùng LLM làm automated judges.

Kết quả thực tế: Flywheel phát hiện rằng task Tool Calling đơn giản đến mức một model fine-tuned llama-3.2-1b-instruct đạt ~98% accuracy so với model 70B đang chạy production.

**Tuy nhiên**, NVIDIA cũng thừa nhận: Flywheel nên được xem như đèn pin soi đường, không phải autopilot — việc promote lên production vẫn là quyết định của con người.

**Nhược điểm:** Yêu cầu NVIDIA GPU, Kubernetes, DGX Cloud — khá nặng cho team nhỏ. Nhưng concept và architecture thì rất đáng học theo.

### 🥈 MLRun (Iguazio) + NVIDIA NeMo

MLRun đã tích hợp với NVIDIA NeMo microservices để mở rộng Data Flywheel Blueprint. Integration này cho phép tự động train, evaluate, fine-tune và monitor AI models ở quy mô lớn.

MLRun đóng vai trò flywheel orchestrator — bao bọc flywheel và điều phối training, fine-tuning theo use case cụ thể, evaluation và monitoring. Nếu cần quyết định từ human-in-the-loop, MLRun sẽ orchestrate feedback đó đến ứng dụng đã deploy.

Đây là combo mạnh nhất hiện tại cho production.

### 🥉 AWS SageMaker + DSPy (Compound AI System)

Amazon xây một framework continuous self-instruct fine-tuning dùng compound AI system, lấy DSPy để điều phối RAG inference, prompt optimization, LLM fine-tuning, và performance evaluation. Flow bắt đầu từ RAG pipeline, qua prompt optimization, rồi sinh synthetic training data để fine-tune, và cuối cùng dùng LLM riêng để tạo feedback cho DPO/PPO alignment.

Approach này hay ở chỗ nó **tự sinh data, tự eval, tự improve** — nhưng cần SageMaker infrastructure.

## Nhóm 2: Orchestration — Xương sống pipeline

Các tools này không train model trực tiếp nhưng **orchestrate toàn bộ flow**: trigger, chạy steps, track artifacts, quyết định promote.

### ZenML

ZenML + OpenPipe integration cho phép khi chạy pipeline lần nữa với data mới, model sẽ tự động được retrain và redeploy, đảm bảo production model luôn phản ánh data mới nhất.

ZenML cho phép viết workflows (pipelines) chạy trên bất kỳ infrastructure backend nào. Nó tự động containerize và track code, track từng run với metrics/logs/metadata, và tích hợp với tools hiện có như MLflow, LangGraph, Langfuse, SageMaker, GCP Vertex.

**Ưu điểm lớn:** Open-source (Apache 2.0), infrastructure-agnostic, có sẵn examples cho LLM fine-tuning pipeline. **Phù hợp nhất với team bạn** vì nhẹ, flexible, chạy được local.

### Dagster

Dagster cung cấp automation pipeline bao gồm refresh data và models theo nhu cầu business, khả năng thử nghiệm hyperparameters khác nhau qua resources, và quản lý environment cho phép test ở dev rồi deploy lên production seamlessly.

Dagster thiên về data engineering hơn nhưng vẫn dùng tốt cho ML pipelines.

### Prefect / Airflow

Traditional orchestrators — vẫn dùng được nhưng không có tính năng ML-native (artifact tracking, model registry) như ZenML hay Dagster.

## Nhóm 3: Specialized — Từng giai đoạn cụ thể

### Data & Labeling

**Argilla** — Argilla là collaboration tool cho AI engineers và domain experts để xây high-quality datasets. Approach lập trình của Argilla cho phép xây workflows cho continuous evaluation và model improvement.

**Distilabel (by Argilla)** — Framework cho synthetic data và AI feedback, tích hợp AI feedback từ bất kỳ LLM provider nào qua unified API. Synthesize và judge data theo latest research papers.

### Evaluation

**DeepEval** — Framework evaluation cho LLM hỗ trợ cả end-to-end và component-level evaluation, với nhiều metrics sẵn dùng powered by LLM, statistical methods, hoặc NLP models chạy local.

**Opik (by Comet)** — Open-source platform cho evaluating, testing, và monitoring LLM applications. Hỗ trợ log traces trong development và production, chạy experiments với different prompts và evaluate against test sets.

### Training

**Unsloth** — Tốt nhất cho Qwen2.5-1.5B (bạn đang dùng), 2x nhanh, 70% ít VRAM.

**Ludwig** — Low-code framework, mọi thứ cấu hình qua YAML declarative — architecture, training loop, hyperparameter search, và backend infrastructure đều có thể chỉnh qua config file. Hỗ trợ LoRA, QLoRA, auto batch size, distributed training.

**torchtune (PyTorch native)** — Library đơn giản hóa quá trình fine-tune và optimization, hỗ trợ distributed training, flexible logging, và model quantization. Có thể apply recipes chỉ với YAML file mà không cần viết code.

### Experiment Tracking

**MLflow** — Model registry, experiment tracking, artifact management. Open-source, mature.

**Weights & Biases** — Experiment tracking, hyperparameter sweeps, model registry. Có free tier.

**Langfuse** — Datasets cho phép tạo test sets và benchmarks để evaluate LLM application, hỗ trợ continuous improvement, pre-deployment testing, và tích hợp với LangChain/LlamaIndex. Bạn đang dùng Langfuse rồi nên đây là lợi thế.

### Step 2: AI Label + Human Label + Agreement

|Task|Tool|Chi tiết|
|---|---|---|
|AI gán nhãn|**Distilabel**|Gọi GPT-4o-mini/Claude Haiku gán nhãn hàng loạt|
|Human review|**Argilla**|UI để labeler xem và gán nhãn|

**Logic agreement bạn mô tả:**

```
IF AI_label == Human_label == User_feedback:
    → auto_approve (high confidence)
ELIF AI_label == Human_label != User_feedback:
    → flag để investigate (có thể user feedback sai)
ELIF AI_label != Human_label:
    → human review lại
```


#### Distilabel (AI Labeling)  + Langfuse để trace lại số tiền

Framework synthetic data/AI feedback, pipeline DAG đơn giản (context manager), tích hợp HF Datasets/LLM (Claude/GPT/OpenAI) unified API. https://github.com/argilla-io/distilabel

- **Dễ dùng**: Quickstart 10 dòng code → load dataset từ Langfuse JSON → TextGeneration step gọi GPT-4o-mini label emotion → push HF Hub.   http://distilabel.argilla.io/1.5.1/sections/getting_started/quickstart/
    Ví dụ emotion labeling:
    

```python
from distilabel.pipeline import Pipeline
from distilabel.steps import LoadDataFromHub, TextGeneration
from distilabel.llms import OpenAI

with Pipeline(name="emotion_labeler") as pipeline:
    load_data = LoadDataFromHub(name="load", dataset="your/langfuse-emotion-logs")
    label = TextGeneration(
        name="label",
        llm=OpenAI(model="gpt-4o-mini"),
        prompt="Classify emotion: {input_text}"
    )
# Run: distiset = pipeline.run()

```

Chạy local/Colab, fault-tolerant, scalable. Thời gian học: 30p docs/tutorial.[](https://argilla-io.github.io/distilabel/1.5.3/sections/getting_started/quickstart/)​
https://argilla-io.github.io/distilabel/1.5.3/sections/getting_started/quickstart/


#### Argilla (Human Review)

UI web + Python SDK cho collaborative labeling, deploy HF Spaces 5p (enable persistent storage).[](https://docs.argilla.io/latest/how_to_guides/)​​

- **Dễ dùng**: `pip install argilla` → `rg.init(api_url, api_key)` → upload Distilabel dataset → UI review (label, confidence, reasoning) → export HF.​

```python
import argilla as rg
```

#### Giải Thích Dễ Hiểu Như "Lego ML"

Hình dung pipeline emotion như dây chuyền nhà máy:

1. **Không ZenML**: Bạn viết 5 script riêng (extract.py, label.py, train.py...), cron chạy theo thứ tự → lỗi 1 step hỏng hết, khó track "run nào tốt nhất?".
    
2. **Có ZenML**: Xây "lego blocks" (@step) → ghép thành pipeline → `zenml run` tự chạy tuần tự/song song, track mọi thứ (data version, F1 score, LoRA file), UI compare "run 1 F1=0.92 vs run 2 F1=0.88".
    

**Vị trí cụ thể trong luồng bạn**:

```text
Raw logs (Langfuse)
  ↓ Step 1: zenml step "extract_data()"  # Pull API → JSON dataset
Labeled data (Distilabel + Argilla)
  ↓ Step 2: zenml step "ai_label()" + "human_review()"
Train data → LoRA model (Unsloth)
  ↓ Step 3: zenml step "finetune_lora()"
Eval metrics (DeepEval)
  ↓ Step 4: zenml step "evaluate_f1()"
Model decision (MLflow)
  ↓ Step 5: zenml step "promote_deploy()" → vLLM canary

```

**Lợi ích đơn giản**:

- **Tự động**: Data mới → trigger rerun toàn pipeline (cron + webhook).
    
- **Track**: UI xem "LoRA run nào tốt nhất?", artifact download.
    
- **Portable**: Chạy local GPU hôm nay → AWS RunPod mai.
    

Ví dụ code siêu ngắn (5p test):

```python
from zenml import step, pipeline

@step  # Lego block 1
def extract() -> str:
    return "emotion_logs.json"  # Từ Langfuse

@step  # Lego 2
def train(logs: str) -> str:
    return "lora_model"  # Unsloth

@pipeline  # Ghép lego
def emotion_flow():
    logs = extract()
    model = train(logs)

emotion_flow()  # Chạy!

```

Với cách bạn đang làm — **lấy data bằng tay từ DataDog, export CSV, chạy script Python evaluate** — thì ZenML là **overkill hoàn toàn** cho giai đoạn này.

## Repo hiện tại làm đúng 1 việc: **Serving**

```
docker-compose.yml
├── vllm-qwen         → Pull image, load model, serve inference
├── emotion-api       → FastAPI proxy, prompt engineering, emotion mapping
├── redis             → Trace queue
└── langfuse-worker   → Push traces lên Langfuse
```

Repo này **không build model, không train model**. Nó nhận 1 model đã có sẵn (`Qwen/Qwen2.5-1.5B-Instruct-AWQ` từ HuggingFace), kéo về, và serve.

Finetune pipeline thì **ngược lại hoàn toàn**: nó nhận data, train model, output ra model artifact mới → cái artifact đó mới được repo serving kéo về serve.

```
FINETUNE REPO                          SERVING REPO (hiện tại)
─────────────                          ────────────────────────
Data → Train → Eval → Model artifact   Model artifact → vLLM → API
         │                                    ▲
         └────────── push model ──────────────┘
                  (HuggingFace Hub / S3 / NFS)
```

Đây là **2 hệ thống khác nhau** nối với nhau qua **model artifact** — không phải qua code.

## 2 repo nối nhau qua gì?

```
EmotionFinetune repo                    VeryFastMoodEmotion repo (hiện tại)
────────────────────                    ────────────────────────────────────

Train xong model
       │
       ▼
Push lên HuggingFace Hub
(hoặc MLflow Model Registry)
       │                                docker-compose.yml:
       │                                  --model org/Qwen2.5-1.5B-Emotion-v2
       │                                           ▲
       └───────────────────────────────────────────┘
                    model artifact
```

Cụ thể: khi finetune xong + promote thành công, Step 5.5 (`publish_model.py`) sẽ:
1. Quantize model mới thành AWQ
2. Push lên HuggingFace Hub (ví dụ: `your-org/Qwen2.5-1.5B-Emotion-v2-AWQ`)
3. Serving repo chỉ cần đổi 1 dòng trong docker-compose:

```yaml
# docker-compose.yml (serving repo)
command: >
  --model your-org/Qwen2.5-1.5B-Emotion-v2-AWQ    # ← đổi model name
```

Hoặc dùng biến environment:

```yaml
command: >
  --model ${EMOTION_MODEL:-Qwen/Qwen2.5-1.5B-Instruct-AWQ}
```

## Thứ duy nhất cần share: Emotion Labels

13 emotion groups cần giống nhau giữa 2 repo. 3 cách:

| Cách | Pros | Cons |
|---|---|---|
| **Python package nhỏ** (`pip install emotion-shared`) | Clean nhất, version control | Overhead maintain package |
| **Config file** (YAML/JSON) trong cả 2 repo | Đơn giản | Phải sync manual |
| **Git submodule** | Auto sync | Git submodule phức tạp |

**Cho Phase 1: copy YAML file là đủ.** 13 emotion groups ít khi thay đổi. Khi nào thay đổi thường xuyên thì upgrade lên shared package.
